Context Full Construction

最終更新: 2026-03-24 18:30 JST
Context Full Construction は、チャンクの短い本文だけでなく、見出し境界まで前後に広げた「文脈つき全文」を組み立てる処理である。この context_full フィールドにより、回答生成時に必要十分な背景情報を LLM へ渡せるようになる。

Context Full Construction は、Docling Hybrid Chunking の後段で各チャンクに付与される context_full フィールドを組み立てる処理です。ドキュメント Ingest Pipeline の中で、チャンク分割によって得られた短い本文だけでは回答生成に必要な背景が不足するため、もとの Markdown 文書から見出し（heading）の境界まで前後に広げたテキストを切り出して添付します。Project Diana Agentic RAG では、この仕組みによって検索精度と回答品質の両立を図っています。

なぜ context_full が必要か

チャンク分割では、検索用のベクトル化に適した短いテキスト（search_text）を作ります。しかし短いテキストだけを LLM（大規模言語モデル）に渡すと、前後の文脈が欠けているために回答が不正確になったり、引用として不自然になったりします。かといって文書全体を渡すとトークン数が膨大になり、コストも応答時間も増えます。

そこで context_full は「短すぎず長すぎない、回答にちょうどよい長さの文脈」を提供します。具体的には、チャンクが属する節（セクション）の見出し境界まで Markdown 原文を広げ、回答生成やファクトチェックに必要な周辺情報をまとめて保持します。この考え方は Search Summary Enrichment の要約付与と補完関係にあり、要約は検索の入口、context_full は回答生成の素材という役割分担です。

組み立てアルゴリズム

context_full の組み立ては、次のステップで行われます。

ステップ1: Markdown 変換と見出し境界の特定

まず、Docling が解析した文書構造から Markdown テキストを生成します。次に、チャンクの元テキストが含まれる位置を特定し、そのテキストを囲む直近の見出し（H1〜H3）の位置を上下に探します。これが context_full の切り出し範囲になります。

ステップ2: 前後拡張とトリミング

見出し境界まで広げたテキストがトークン上限を超える場合は、文の区切り（句点やピリオド）で短くします。このとき、チャンク本文を含む中心部分は必ず残し、前後を対称的に削ります。Chunk Oversize Splitting と同様にトークン数ベースで判定するため、マルチバイト文字の多い日本語文書でも安定して動作します。

ステップ3: メタデータの付与

完成した context_full には、元の見出し階層（search_headings）とページ番号が付与されます。これにより、回答生成時に「どの章・節から引用したか」をたどれるようになります。

Context Full Construction の流れ: Markdown 文書からチャンクを囲む見出し境界を特定し、トークン上限内でトリミングして context_full フィールドを組み立てる。

トークン予算と切り詰め戦略

context_full のトークン予算は、Policy-Enforced Retrieval が回答生成時に渡すプロンプトの最大長から逆算されます。一般的には 1 チャンクあたり 1,000〜2,000 トークン程度が割り当てられます。

予算を超えた場合の切り詰めには、いくつかのルールがあります。まず、チャンク本文（search_text の元テキスト）は最優先で保持されます。次に、本文の前後のテキストを文単位で対称的に削ります。最後に、それでも超える場合は前方（見出し直後のテキスト）を優先し、後方を多めに削ります。これは、節の冒頭には定義や導入が書かれていることが多く、回答の正確性に寄与しやすいためです。

この予算管理は Retrieval Budgeting の一部として設計されており、検索ラウンドごとに使えるトークン総量と連動しています。

検索・回答生成との関係

context_full は検索そのものには使われません。検索には search_text（Search Summary Enrichment で生成される要約）がベクトル化されて使われます。context_full が活躍するのは、検索でヒットしたチャンクを LLM に渡して回答を生成する段階です。

つまり、検索の入口では短い要約でマッチングし、回答の生成では広い文脈で正確性を担保するという二段構えになっています。この設計により、検索精度（recall）と回答品質（faithfulness）を両立しています。Retrieval Round Policy が複数ラウンドの検索を制御するとき、各ラウンドで返されるチャンクの context_full が回答のエビデンスとして蓄積されます。

エッジケースへの対処

見出しがない文書

Docling Hybrid Chunking では、見出しがない文書には synthetic heading（合成見出し）を自動付与します。context_full の組み立てでもこの合成見出しを境界として利用するため、見出しのない文書でも安定して動作します。

テーブルや画像が含まれるチャンク

表を含むチャンクでは、表の前後の見出し境界まで context_full を広げると非常に長くなることがあります。この場合、表の Markdown 表現はそのまま残しつつ、前後の段落テキストを優先的に削ります。Picture OCR Stage で付与された画像テキストは context_full には含めず、別フィールド（picture_texts）として保持します。これにより、context_full のトークン予算を圧迫しません。

Oversize split 後のチャンク

Chunk Oversize Splitting で追加分割されたチャンクについては、分割前の元チャンクと同じ見出し境界を共有します。ただし、context_full の内容は各分割チャンクの本文位置に応じて再計算されるため、同じ節内でも分割チャンクごとに異なる context_full が生成されます。

実装上のポイント

context_full の組み立ては job_src.chunking モジュール内で、チャンク enrichment の一部として実行されます。Markdown 変換には Docling の export_to_markdown() を利用し、見出し境界の検出は正規表現で ^#{1,3}\s パターンをマッチさせます。

トークン数の計算には、Split Deployment RAG の各モジュールで共通の tokenizer（tiktoken）が使われます。これは 環境変数コントラクト管理 で定義されたモデル名から自動選択されるため、モデルを変更してもトークン計算が一貫します。

将来的には、見出し境界だけでなく意味的な段落境界（Semantic Paragraph Segmentation）を利用した、より精密な文脈切り出しが検討されています。また、複数チャンクの context_full を統合して重複を排除する Context Deduplication Merge や、文書種別に応じて切り出し戦略を変える Document-Type Aware Context も今後の課題です。
