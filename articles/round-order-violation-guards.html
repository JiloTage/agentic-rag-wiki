Round Order Violation Guards

最終更新: 2026-03-24 19:00 JST
Round Order Violation Guards は、Retrieval Round Policy が定める検索ラウンドの進行順序（mandatory_broad → optional_narrowing → optional_relaxed）を飛ばせないようにするガード（安全装置）である。allowed_next テーブルに基づいて次に許可される段階を制限し、順序を逸脱した遷移は policy_order_violation として検知され、その検索は失敗扱いになる。

Round Order Violation Guards は、Project Diana Agentic RAG の検索処理において、ラウンド（検索の段階）の順番を厳密に守らせる仕組みである。Policy-Enforced Retrieval（ポリシーに従って検索を制御する設計）の一部として動作し、mandatory_broad（広く探す段階）→ optional_narrowing（絞り込む段階）→ optional_relaxed（条件をゆるめる段階）という決められた順番を飛ばすことを許さない。たとえば「結果が少ないから最初から条件をゆるめよう」という場当たり的な判断をモデルが行っても、このガードが介入して policy_order_violation（順序違反）として検出・拒否する。

ガードの仕組み

ガードの核となるのは allowed_next テーブルである。これは各ラウンド段階に対して「次に遷移してよい段階」を定義した対応表で、以下のような構造を持つ。

  • None（初期状態）→ mandatory_broad のみ許可
  • mandatory_broad → mandatory_broad または optional_narrowing を許可
  • optional_narrowing → optional_narrowing または optional_relaxed を許可
  • optional_relaxed → optional_relaxed のみ許可（後戻りも不可）

検索の各ラウンドが開始される前に、resolve_round_policy() が現在の段階と次に要求された段階を allowed_next テーブルと照合する。一致しなければ即座に policy_order_violation を発行し、そのラウンドの検索は実行されない。このチェックは Retrieval Round Policy の resolve_round_policy() 内で最初に実行されるため、予算計算やフィルタ構築よりも先に順序の正当性が確認される。

allowed_next テーブルに基づくラウンド遷移図。青い実線は許可される遷移、赤い破線は policy_order_violation として拒否される遷移を示す。

違反の検知と記録

順序違反が検知されると、ガードは以下の情報を validation_log（検証ログ）に記録する。

  • violation_type: "policy_order_violation" という固定の種別
  • current_phase: 違反が発生した時点のラウンド段階
  • requested_phase: 要求された（許可されなかった）ラウンド段階
  • allowed_phases: その時点で許可されていた段階の一覧
  • round_index: 何回目の検索で違反が起きたか

この記録は Workspace Trace UI（トレース表示画面）で確認できる。運用チームは、違反が頻発している場合に Retrieval Round Policy の設定や、モデルのプロンプト設計を見直す手がかりにする。また、Violation Alert Routing（違反アラートの配信ルール）を設定すれば、特定の違反パターンが閾値を超えた際に通知を受け取ることもできる。

失敗扱いの挙動

policy_order_violation が発行された検索ラウンドは、結果を返さずに「失敗」として扱われる。ただし、検索プロセス全体が即座に中止されるわけではない。ToolRoundExecutor は違反が起きたラウンドをスキップし、残りの検索予算がまだあれば次のラウンドに進むことを試みる。

ここで重要なのは、違反後の「次のラウンド」が allowed_next テーブル上で正当な遷移である場合のみ実行されるということである。たとえば、mandatory_broad から optional_relaxed への飛び越しが拒否された場合、次に optional_narrowing を要求すればそれは正当な遷移として許可される。

検索予算の管理については Retrieval Budgeting（検索予算の配分）を、違反発生時の予算消費ルールについては Violation Budget Accounting（違反時の予算計上ルール）を参照されたい。違反で消費された予算を回復するかどうかは Optimization Priority Modes（最適化の方針モード）の設定に依存する。

環境変数による制御

ガードの挙動は環境変数で細かく制御できる。主な設定項目は以下のとおりである。

  • ROUND_GUARD_STRICT_MODE: true の場合、違反が 1 回でも発生すると以降のラウンドをすべて中止する（デフォルトは false）
  • ROUND_GUARD_LOG_LEVEL: 違反ログの詳細度を minimal / standard / verbose から選択する
  • ROUND_GUARD_ALLOWED_OVERRIDES: 特定のテスト環境で遷移制約を一部緩和するための設定（本番環境では無効化が推奨される）

これらの環境変数は 環境変数コントラクト管理（設定の一元管理）で contract.json に定義され、Env Consistency Checks（環境変数の整合性検証）によってプロファイル間の不整合がないか検証される。間違った設定でガードが無効化されると順序制約の意味がなくなるため、CI パイプラインでの検証が不可欠である。

テストとデバッグ

ガードの正しさを確認するために、以下のテスト戦略が用いられる。

ユニットテスト

allowed_next テーブルの全遷移パターン（許可・拒否の両方）を網羅するテストを用意する。特に境界ケース（同じ段階への繰り返し、最終段階からの遷移など）を重点的にテストする。

統合テスト

実際の検索フローを模した統合テストで、ガードが ToolRoundExecutor と正しく連携するかを検証する。違反発生時にログが正しく記録されるか、後続のラウンドが適切にスキップまたは実行されるかを確認する。

トレースによるデバッグ

Workspace Trace UI のラウンド表示で、各ラウンドの遷移判定結果をリアルタイムに確認できる。違反が発生した場合は赤色で強調表示され、allowed_phases と requested_phase の差分が一目でわかる。Guard Replay Simulator（ガード再現シミュレータ）を使えば、過去のセッションのラウンド遷移を再現して原因を特定することもできる。

設計思想

このガードが存在する最大の理由は、「検索の質よりも先に、ルールどおりに動いているかを守る」という方針にある。LLM（大規模言語モデル）が検索条件を自律的に決める Project Diana Agentic RAG のような仕組みでは、モデルが効率を優先して段階を飛ばす判断をすることがある。しかし、Policy-Enforced Retrieval の設計思想では、mandatory filter（必須フィルタ）を飛ばすことは検索結果のブレではなくルール違反に直結する。

Round Order Violation Guards は、この「ルール優先」の原則を実行時に強制する最後の砦である。ドキュメント Ingest Pipeline（文書取り込みパイプライン）で付与されたメタデータの品質がばらついていても、ガードが順序を守らせることで、少なくとも「広く探してから絞り込む」という基本戦略が崩れないことを保証する。

将来的には Adaptive Guard Relaxation（適応的なガード緩和）のように、過去の実績に基づいて段階のスキップを安全に許可する仕組みも検討されているが、現時点では厳格な順序制約が維持されている。
