環境変数コントラクト管理

最終更新: 2026-03-24 17:53 JST
環境変数コントラクト管理は、`env/contract.json` を source of truth として chat/job/web の設定責務を統一する運用モデルである。Project Diana では profile 生成、整合性検証、runtime access convention がこの契約を中心に組まれている。

環境変数コントラクト管理 とは、複数モジュールにまたがる env key の意味、所有者、必須性、profile ごとの上書きを単一の契約ファイルで管理する考え方である。Project Diana Agentic RAG では split deployment を支える基盤として env/contract.json が置かれ、Split Deployment RAG を運用可能にする接着層になっている。

Source of Truth の置き方

ドキュメント docs/env-unified-management.md は、source of truth を三層で説明している。第一に env/contract.json が契約レジストリであり、第二に各 module の env catalog が runtime から参照されるキー定義であり、第三に apps/<module>/.env.example が base template になる。意図された関係は「catalog == base template == contract の rule union」であり、設定ファイルをただ増やすのではなく、三者の一致を維持すること自体が設計目標になっている。

この契約では key ごとに owner、domain、説明文が定義され、module 側では required、non_local_required、optional、template で分類される。つまり env の一覧表ではなく、運用ルールと責任分担を持ったメタデータ層として機能している。

env contract は catalog と template と profile override を束ね、生成済み `.env.<profile>` を通して runtime process environment をそろえる。

契約ファイルの構造

Key metadata

env/contract.json の keys セクションには、個々の env key の説明が日本語で書かれている。ここでの説明は placeholder を禁じられており、単にキー名を繰り返すのではなく用途を説明しなければならない。これは運用者が設定値の意味を誤読しないための guard であり、設定ファイルの自己文書化を強制している。

Module rules

rules では module ごとに required / non-local-required / optional / template を分ける。たとえば chat は JWT_SECRET や INGEST_WEBHOOK_SECRET を扱い、web は公開可能な NEXT_PUBLIC_* 系だけを使う。このルール分離により、どの値がどの実行単位に必要かが明確になる。

Profile overrides

profiles.modules.<module>.<profile> には local、dev、stg、prod の上書きが入り、render 時に base template へ流し込まれる。ここで profile override は base template の部分集合でなければならず、未知キーは許されない。この制約は Env Profile Rendering と Env Consistency Checks の基礎になる。

レンダリングと配布

tools/render_env_profiles.py は、選択した module と profile に対し、.env.example を読み込み、contract の override を差し込んで .env.<profile> を出力する。処理は単純だが重要で、profile 側が base template に存在しないキーを持つ場合はエラーで止まる。つまり render 処理そのものが drift 検知の第一段階を兼ねている。

この生成フローは split deployment で特に効く。chat、job、web が別々に起動される以上、手作業コピーで env を配るとすぐに齟齬が出る。render スクリプトを経由させることで、全モジュールが同じ契約に従って設定される。

整合性検証

tools/check_env_consistency.py は、contract、catalog、base template、profile override の一致を検証する。domain index が metadata と一致するか、各 profile が APP_ENV または NEXT_PUBLIC_APP_ENV を正しく設定しているか、required key に空値が残っていないかまで確認する。このスクリプトがあることで、設定ミスはデプロイ後ではなく CI や local check で発見される。

README と docs はさらに make env-check、python tools/check_env_convention.py、python tools/check_env_consistency.py をセットで回すよう案内している。したがって env contract は「読むだけの仕様書」ではなく、コマンドで実際に強制される契約である。

実装上の access convention

Project Diana では Python サービスが business module から直接 os.getenv() を呼ぶことを避け、module env alias を介して読む。web 側でも process.env の直接参照を限定している。これは値の取得地点を統一し、後からキーの棚卸しやテスト override をしやすくするための設計である。

この lazy read 方針は、import 時に env が確定していなくてもテストを壊しにくい。したがって env contract 管理は deployment だけの話ではなく、実装スタイルそのものにも影響している。さらに掘るなら Module Env Catalogs や Lazy Env Access Pattern が派生記事になる。
