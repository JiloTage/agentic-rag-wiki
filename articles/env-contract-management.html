環境変数コントラクト管理

最終更新: 2026-03-24 18:08 JST
環境変数コントラクト管理とは、env/contract.json という1つのファイルに、chat・job・web の設定情報をまとめて書いておくやり方です。このファイルが「正しい情報のもと」になります。Project Diana では、設定ファイルの作成、設定の正しさチェック、コードでの設定値の読み方まで、すべてこのファイルをもとに決めています。

環境変数コントラクト管理 は、chat、job、web などのモジュール（機能のまとまり）が使う環境変数（アプリの動作を決める設定値）を、1つの契約ファイルで管理する考え方です。「この設定値は何のためにあるのか」「誰が管理するのか」「必須なのか」「環境ごとにどう変えるのか」を、すべてこのファイルに書きます。Project Diana Agentic RAG では、env/contract.json がその契約ファイルです。Split Deployment RAG（サービスを分けて動かす構成）をうまく動かすための「つなぎ役」になっています。

Source of Truth の置き方

ドキュメント docs/env-unified-management.md では、「正しい情報のもと（Source of Truth）」を3つに分けて説明しています。1つ目は env/contract.json で、すべての設定値のルールを決めるファイルです。2つ目は各モジュールの env catalog（そのモジュールが使う設定値の一覧）です。3つ目は apps/<module>/.env.example で、設定ファイルのひな型です。大事なのは、この3つの内容がいつも合っていることです。設定ファイルをバラバラに作るのではなく、3つをそろえておくことがこの設計の目的です。

契約ファイルには、環境変数ごとに owner（誰が管理するか）、domain（どの分野か）、説明文が書いてあります。モジュール側では、各設定値を required（必ず必要）、non_local_required（ローカル以外で必要）、optional（あってもなくてもよい）、template（ひな型の値）に分けています。こうすることで、「誰が何を管理するか」がはっきりわかるようになっています。

env contract（契約ファイル）が catalog、template、profile override の情報をまとめます。そこから .env.<profile> が生成され、各サービスが同じ設定で動けるようになります。

契約ファイルの構造

Key metadata

env/contract.json の keys セクションには、それぞれの環境変数の説明が日本語で書いてあります。説明文にはキー名をそのまま繰り返すだけではなく、「この値は何に使うのか」を書く決まりです。こうしておくと、設定ファイルを読むだけで意味がわかるので、設定ミスを防ぎやすくなります。

Module rules

rules セクションでは、モジュールごとに「必ず必要な値」「ローカル以外で必要な値」「あってもなくてもよい値」「ひな型の値」を分けて書きます。たとえば chat モジュールは JWT_SECRET や INGEST_WEBHOOK_SECRET を使います。web モジュールは公開しても問題ない NEXT_PUBLIC_* の変数だけを使います。こうして分けておくと、「どのサービスにどの設定が必要か」がすぐにわかります。

Profile overrides

profiles.modules.<module>.<profile> には、環境ごとに変えたい値が入っています。環境は local（手元の開発用）、dev（開発サーバー）、stg（本番前のテスト用）、prod（本番）の4つです。設定ファイルを作るとき、ひな型の値をこの上書き値に置きかえます。ひな型にないキーを上書きに書くことはできません。このルールが Env Profile Rendering や Env Consistency Checks の仕組みのもとになっています。

レンダリングと配布

tools/render_env_profiles.py は、モジュールと環境を指定して実行するスクリプトです。.env.example（ひな型）に契約ファイルの上書き値を当てはめて、.env.<profile> を出力します。もしひな型にないキーが上書き値に含まれていたら、エラーで止まります。つまり、設定ファイルを作る時点で「設定のずれ」に気づけます。

この仕組みは、サービスを分けて動かす構成で特に役立ちます。chat、job、web がそれぞれ別々に起動するため、手作業で設定ファイルをコピーするとすぐに食い違いが起きます。この生成スクリプトを使えば、すべてのモジュールが同じルールに沿った設定ファイルを持てます。

整合性検証

tools/check_env_consistency.py は、契約ファイル、catalog、ひな型、上書き値がちゃんと合っているかを調べるスクリプトです。たとえば、分野の分類が正しいか、各環境で APP_ENV や NEXT_PUBLIC_APP_ENV が正しいか、必ず必要な値が空のままになっていないか、などを確認します。このスクリプトがあるので、設定ミスは本番にデプロイする前に見つけられます。

README やドキュメントでは、make env-check、python tools/check_env_convention.py、python tools/check_env_consistency.py をセットで実行するように書かれています。つまり env contract は「読むだけの書類」ではなく、コマンドで実際にチェックできる「生きたルール」です。

実装上の access convention

Project Diana では、Python のコードから直接 os.getenv() を呼ぶのではなく、モジュール専用の窓口（env alias）を通して設定値を取り出します。web 側でも process.env を直接使う場所を決めています。こうすると、設定値を読む場所が1か所にまとまるので、あとからキーの確認やテスト用の値の入れ替えがしやすくなります。

もう1つよい点があります。「使うときに初めて読む（lazy read）」方針なので、テストのときに環境変数がまだ決まっていなくても問題が起きにくいのです。このように、env contract の管理はデプロイだけでなく、コードの書き方にも関わっています。もっと詳しく知りたい場合は、Module Env Catalogs や Lazy Env Access Pattern をご覧ください。
