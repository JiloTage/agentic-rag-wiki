Env Consistency Checks

最終更新: 2026-03-24 18:08 JST
Env Consistency Checks は、Project Diana の tools/check_env_consistency.py が env/contract.json、module catalog、.env.example、profile override の一致を検証する仕組みである。requiredness、owner/domain、domain_index、profile ごとの APP_ENV を静的に突き合わせ、設定 drift を CI と local check で止める。

Env Consistency Checks は、環境変数コントラクト管理 を単なる文書ではなく実行可能な契約に変える検証レイヤーである。Project Diana では chat、job、web の三モジュールに対して、catalog、template、contract、profile override を突き合わせ、Split Deployment RAG 全体で設定 drift が広がる前に止める。

何を照合するか

tools/check_env_consistency.py は module を chat、job、web に固定し、profile を local、dev、stg、prod に固定している。照合対象は env/contract.json、各 module の env catalog、apps/<module>/.env.example、profile override であり、意図された関係は「catalog == base template == contract rule union」である。

contract、catalog、base template、profile override の四点を同時に突き合わせることで、単発のファイル比較では見えない drift を止める。

contract 側の制約

check script は domain を auth、retrieval、storage、observability、runtime の五つに固定し、domain_index のキー順と内容が metadata の domain assignment と一致することまで確認する。また key description は空欄禁止で、TODO や低情報量の placeholder も reject される。つまり contract には「キーがある」だけでなく「意味が説明されている」ことまで要求される。

この層は Domain Index Integrity や Env Owner Domain Rules のトピックに分解できる。設定の正しさは値だけでなく、owner と domain のメタデータが成立しているかにも依存するからである。

module ごとの整列

各 module について script は「base template にあるキーが catalog にもあるか」「catalog のキーが contract rules に宣言されているか」「runtime contract key の owner が shared または当該 module か」を見る。たとえば web は NEXT_PUBLIC_APP_ENV を使い、chat / job は APP_ENV を使うため、同じ環境概念でも module ごとに見るキーが異なる。

この比較により、catalog の更新漏れや template への追記漏れがすぐ分かる。周辺テーマとしては Module Env Catalogs と ドキュメント Ingest Pipeline の job module 側 runtime contract がある。

profile ごとの必須性

profile override は base template の部分集合でなければならず、未知キーは即座に invalid になる。また各 profile は chat / job では APP_ENV=<profile>、web では NEXT_PUBLIC_APP_ENV=<profile> を必ず持たなければならない。さらに required は全環境で非空、non_local_required は dev / stg / prod で非空であることがチェックされる。

このため render 済み env file の正当性は、生成後ではなく生成前の contract レベルで担保される。ここから Env Profile Rendering、Rendered Runtime Env Files、Check-Mode Drift Detection、Profile Requiredness Matrix といった派生が生まれる。

開発フローへの組み込み

docs/env-unified-management.md は、root で python tools/check_env_convention.py と python tools/check_env_consistency.py を回し、必要に応じて make env-check と render を続ける流れを案内している。つまり consistency check は CI だけの道具ではなく、ローカル作業開始前の gate として設計されている。

この gate が効く範囲は広い。Policy-Enforced Retrieval の env key や、Retrieval Round Policy の optimize priority、さらに Azure Container Apps Topology で配る secret / table 名の正しさも、最終的には env contract と consistency check に戻ってくる。

何を未然に防ぐか

この仕組みが止めるのは、単純な typo だけではない。module owner の取り違え、profile ごとの空値、domain index の未更新、template と catalog の片方だけが変わった状態など、後から発見すると原因追跡が難しい drift を先に表面化させる。split deployment では chat / job / web が別起動単位なので、こうした齟齬が本番で起きると調査コストが急増する。

その意味で Env Consistency Checks は、設定ファイル版の static analysis に近い。実際の runtime へ進む前に contract レベルで止めるからこそ、lazy env access や profile rendering が安心して使える。
