Env Consistency Checks

最終更新: 2026-03-24 18:08 JST
Env Consistency Checks は、環境変数（アプリの設定値）がファイルの間でずれていないかを自動でチェックする仕組みです。tools/check_env_consistency.py というスクリプトが、設定のルール、変数の一覧、テンプレート、環境ごとの値を見くらべて、おかしいところを教えてくれます。

Env Consistency Checks は、環境変数コントラクト管理（環境変数のルールを決めて管理する仕組み）を、ドキュメントだけでなくスクリプトでも確認できるようにしたものです。chat、job、web の3つのモジュールについて、変数の一覧・テンプレート・ルール・環境ごとの設定を見くらべます。Split Deployment RAG のどこかで設定がずれていたら、すぐにわかります。

何を照合するか

tools/check_env_consistency.py は、chat、job、web の3つのモジュールと、local、dev、stg、prod の4つの環境を対象にしています。チェックするのは次の4つです。(1) env/contract.json（ルールの定義）、(2) 各モジュールが使う変数の一覧、(3) apps/<module>/.env.example（テンプレート）、(4) 環境ごとの上書き設定。この4つの間で食い違いがないかを調べます。

ルール、変数一覧、テンプレート、環境別設定の4つを見くらべます。1つずつ見ただけでは気づけないズレを見つけます。

contract 側の制約

スクリプトは、変数の分類（domain）を auth、retrieval、storage、observability、runtime の5つだけにしています。domain_index の順番と中身が、各変数についた分類と合っているかも調べます。変数の説明は空にできません。TODO のような仮の説明もダメです。「変数がある」だけでなく、「何のための変数か」がわかるようにする必要があります。

くわしくは Domain Index Integrity や Env Owner Domain Rules をご覧ください。設定が正しいかどうかは、値だけでなく「誰が管理しているか」「どの分類か」も大事です。

module ごとの整列

モジュールごとに3つを調べます。「テンプレートにある変数が一覧にもあるか」「一覧の変数がルールにも登録されているか」「その変数の管理者は正しいか」です。たとえば、web は NEXT_PUBLIC_APP_ENV を使います。chat と job は APP_ENV を使います。同じ意味の値でも、モジュールによって名前が違います。

このチェックがあると、「一覧に変数を足したのにテンプレートに書き忘れた」というミスにすぐ気づけます。関連する記事として Module Env Catalogs や、ドキュメント Ingest Pipeline の job 側の設定があります。

profile ごとの必須性

環境ごとの設定では、テンプレートにある変数だけを使えます。テンプレートにない変数を書くとエラーになります。chat と job では APP_ENV=<profile>、web では NEXT_PUBLIC_APP_ENV=<profile> が必ず要ります。「必須」の変数はすべての環境で値が入っている必要があります。「ローカル以外で必須」の変数は、dev、stg、prod で値が必要です。

こうすると、環境変数ファイルを作る前に、設定が正しいか確認できます。関連する記事として Env Profile Rendering、Rendered Runtime Env Files、Check-Mode Drift Detection、Profile Requiredness Matrix があります。

開発フローへの組み込み

docs/env-unified-management.md に、開発を始める前の手順が書いてあります。まず python tools/check_env_convention.py と python tools/check_env_consistency.py を実行します。問題があれば make env-check でファイルを作り直します。このチェックは CI（自動テスト）だけでなく、ローカルで開発するときにも使います。

チェックの対象は広いです。Policy-Enforced Retrieval の変数、Retrieval Round Policy の設定値、Azure Container Apps Topology の秘密情報やテーブル名なども、このチェックで正しさを確認します。

何を未然に防ぐか

このチェックは、タイプミスだけでなく、いろいろな問題を防ぎます。たとえば「変数の管理者が間違っている」「ある環境だけ値が空」「分類の一覧を更新し忘れた」「テンプレートだけ変えて一覧を変え忘れた」などです。chat、job、web は別々に動くので、こうしたズレが本番で起きると原因を探すのがとても大変です。

Env Consistency Checks は、プログラムを動かす前に設定のミスを見つける仕組みです。この段階で問題を止められるので、安心して環境変数ファイルを作れます。
