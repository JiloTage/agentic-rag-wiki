Domain Index Integrity

最終更新: 2026-03-24 19:00 JST
Domain Index Integrity は、env/contract.json の domain_index（ドメイン分類の索引）が、各環境変数に付与された key metadata（キーごとのメタ情報）と一致していることを強制する検証設計である。auth、retrieval、storage、observability、runtime の5ドメインについて、索引の順番と内容が各変数の分類と整合しているかを静的にチェックする。

Domain Index Integrity は、環境変数コントラクト管理の中核ルールの一つである。env/contract.json には各環境変数のメタ情報（owner、domain、requiredness など）が定義されているが、それとは別に domain_index というセクションがある。このセクションは「どのドメインにどの変数が属するか」を索引として一覧化したものである。Env Consistency Checks の中で、この索引と個々の変数の domain フィールドが矛盾していないかを検証するのが Domain Index Integrity の役割である。

5つのドメイン

Project Diana では環境変数を次の5つのドメインに分類している。

  • auth — 認証・認可に関する変数（APIキー、トークン、シークレットなど）
  • retrieval — 検索・取得に関する変数（Policy-Enforced Retrieval の設定値、インデックス名など）
  • storage — データ保存に関する変数（接続文字列、テーブル名、Azure Container Apps Topology のストレージ設定など）
  • observability — 可観測性に関する変数（ログレベル、トレースエンドポイント、サンプリング率など）
  • runtime — 実行環境に関する変数（APP_ENV、ポート番号、ワーカー数など）

この5分類は Env Owner Domain Rules で詳しく定義されている。domain_index はこの5つのドメインをキーとし、それぞれに属する変数名の配列を値として持つ。

domain_index の構造

domain_index は contract.json のトップレベルに置かれる辞書（オブジェクト）である。キーはドメイン名、値は変数名の配列で、各配列はアルファベット順にソートされている必要がある。

domain_index の各エントリと key metadata の domain フィールドが双方向に一致しているかを検証する。片方にしかない変数や分類の不一致はエラーとなる。

検証ルール

Domain Index Integrity の検証は、tools/check_env_consistency.py の中で以下の4つのルールとして実装されている。

完全性チェック

contract.json の keys セクションに定義されているすべての変数が、domain_index のいずれかのドメイン配列に含まれている必要がある。逆に、domain_index に含まれる変数名はすべて keys セクションに存在しなければならない。片方にしかない変数があればエラーとなる。

一貫性チェック

各変数の key metadata に記載された domain フィールドの値と、その変数が domain_index で所属しているドメインが一致していなければならない。たとえば、ある変数の domain が auth なのに domain_index では storage の配列に入っていたらエラーである。

ソート順チェック

各ドメインの変数配列はアルファベット昇順でソートされている必要がある。順番が崩れていると、差分比較（diff）が見づらくなり、レビュー負荷が増える。Check-Mode Drift Detection でも、ソート済み配列であることを前提にした差分検出が行われる。

ドメイン集合チェック

domain_index のキー集合は、許可された5ドメイン（auth, retrieval, storage, observability, runtime）と完全に一致しなければならない。未知のドメインが追加されていたり、既存のドメインが欠落していたりするとエラーとなる。これは Env Owner Domain Rules で定められた分類体系を守るための制約である。

典型的な違反パターン

開発中によく発生する違反パターンをいくつか紹介する。

  • 変数追加の片手落ち — keys セクションに新しい変数を追加したが、domain_index への追記を忘れた。Module Env Catalogs と contract の同期漏れで起きやすい。
  • ドメイン変更の不整合 — 変数のドメインを runtime から observability に変更したが、domain_index の移動を忘れた。
  • ソート崩れ — 手動編集で変数を配列の末尾に追加し、アルファベット順を壊した。
  • 変数削除の残骸 — keys セクションから変数を削除したが、domain_index に名前が残っている。

これらはすべて Env Consistency Checks の実行時に検出され、CI（継続的インテグレーション）やローカルの make env-check で開発者にフィードバックされる。

他の検証との連携

Domain Index Integrity は単独で動くのではなく、Env Consistency Checks の一部として、他の検証と組み合わせて実行される。

  • Profile Requiredness Matrix — 必須性の検証。domain_index が正しくないと、ドメイン単位での必須性レポートも不正確になる。
  • Env Profile Rendering — 環境変数ファイルの生成。domain_index を参照してドメインごとのコメントブロックを生成する場合がある。
  • Rendered Runtime Env Files — 生成されたファイルの検証。domain の整合性が壊れていると、生成結果もおかしくなる。
  • Domain Index Migration Tooling — ドメイン分類の変更を安全に行うための移行ツール。

Split Deployment RAG の chat、job、web の3モジュールは、同じ contract.json を参照するため、domain_index の不整合はすべてのモジュールに影響する。Project Diana Agentic RAG の運用安定性にとって、この検証は基盤的な役割を果たしている。

なぜ索引を別に持つのか

各変数に domain フィールドがあるなら、domain_index は冗長に見えるかもしれない。しかし索引を別に持つ理由がある。

  • 一覧性 — ドメインごとにどの変数が属しているかを、keys セクション全体を走査せずに把握できる。変数が100個を超えると、この一覧性は重要になる。
  • レビュー効率 — PR（プルリクエスト）で domain_index の差分を見れば、どのドメインに変数が増減したかが一目でわかる。
  • ツール連携 — Env Profile Rendering や Contract Schema Linter などのツールが、ドメインごとの変数リストを高速に参照できる。
  • 整合性の二重保証 — 冗長だからこそ、片方を変え忘れたときに検出できる。これは意図的な設計である。

この「意図的な冗長性」のアプローチは、Cross-Reference Redundancy Pattern として他のシステムでも見られる設計パターンである。
