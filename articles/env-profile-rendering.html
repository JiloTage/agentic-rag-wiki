Env Profile Rendering

最終更新: 2026-03-24 18:08 JST
Env Profile Rendering は、テンプレート（ひな形）とプロファイル（環境ごとの設定）を合わせて、アプリが使う設定ファイル（`apps//.env.`）を自動で作る仕組みです。Project Diana では、chat・job・web の 3 つのアプリに設定を届けるために `tools/render_env_profiles.py` というスクリプトを使います。

Env Profile Rendering は、環境変数コントラクト管理 で決めたルールをもとに、アプリが実際に使う設定ファイルを作る処理です。Project Diana では、tools/render_env_profiles.py というスクリプトが chat・job・web のひな形ファイルを読み込みます。そして、プロファイル（本番用・開発用などの環境ごとの設定）に合わせて値を書き換え、.env.local のようなファイルを出力します。

入力は 3 つある

この処理には 3 つの入力があります。1 つ目は、コントラクト（「どの設定項目を、どんな値に変えてよいか」を決めたルール）です。2 つ目は、ベーステンプレート（apps/chat/.env.example、apps/job/.env.example、apps/web/.env.example）です。3 つ目は、どのアプリのどのプロファイルを使うかの指定です。コントラクトで許可された項目だけが変わるので、想定外の値が入り込む心配がありません。

この仕組みのおかげで、Split Deployment RAG でも chat・job・web に同じ方法で設定を届けられます。ひな形をまるごとコピーするのではなく、各アプリの元ファイルはそのまま残して、環境ごとに違う部分だけを上書きします。

コントラクト（ルール）とベーステンプレート（ひな形）を合わせて `.env.` ファイルを作り、各アプリの起動時に必要な設定を届けます。

render スクリプトがしていること

スクリプトは、まずコントラクトを読み込みます。次に、指定されたアプリとプロファイルを確認します。そしてひな形ファイルを 1 行ずつ見ていき、該当する設定項目の値だけを書き換えます。もしひな形に存在しない項目を書き換えようとすると、エラーになります。これにより、設定の間違いを早めに見つけられます。

--check モードを使うと、ファイルを作らずに、今ある .env.<profile> が最新の内容と合っているかだけを調べられます。CI（自動テストの仕組み）で「設定が古くなっていないか」を確認するのに便利です。くわしくは Check-Mode Drift Detection をご覧ください。

モジュール差分の扱い

chat と job は Python で書かれており、検索・OpenAI・ストレージなど、たくさんの設定項目があります。一方、web は NEXT_PUBLIC_API_BASE_URL と NEXT_PUBLIC_APP_ENV くらいしかありません。render スクリプトは、この違いを無理に 1 つにまとめません。アプリごとにひな形は別々のまま、共通の処理だけをまとめています。

この作りは、各アプリの役割が違うことをそのまま表しています。Module Env Catalogs を見ると、chat と job では必要な設定がかなり違うのに、同じ render の手順で処理できていることがわかります。

運用上の意味

Project Diana では、ローカルでアプリを動かす前に python tools/render_env_profiles.py --module ... --profile local を実行します。アプリは load_dotenv() を使わず、このスクリプトで作ったファイルから設定を読み込む仕組みだからです。つまり render は「あると便利なツール」ではなく、起動に必要な手順の一部です。

また、Policy-Enforced Retrieval（ポリシーに従った検索の制御）で使う設定や、top-k（検索結果を上位何件まで使うかの数）も、この render で環境ごとに用意されます。そのため、env profile rendering は設定を配るだけでなく、アプリの動き方を切り替える入口にもなっています。

次に読むべき周辺

もっとくわしく知りたい方は、次の記事がおすすめです。ルールとひな形がちゃんと合っているかを確認する Env Consistency Checks。アプリごとの設定項目の違いをまとめた Module Env Catalogs。生成されたファイルそのものについて書かれた Rendered Runtime Env Files。ローカル開発については Local Docker Compose Profile も関連します。
