Env Profile Rendering

最終更新: 2026-03-24 18:08 JST
Env Profile Rendering は、base template と contract profile override から `apps//.env.` を生成する仕組みである。Project Diana では split deployment の各モジュールへ設定を配る共通経路として `tools/render_env_profiles.py` が使われる。

Env Profile Rendering は、環境変数コントラクト管理 を実際の runtime file へ落とすためのスクリプト処理である。Project Diana では tools/render_env_profiles.py が chat、job、web の base template を読み、profile override を適用して .env.local などの実行用ファイルを生成する。

入力は 3 つある

この render 処理が依存するのは、contract、base template、module/profile の選択である。contract からは上書き値を読み、apps/chat/.env.example、apps/job/.env.example、apps/web/.env.example が置換対象になる。つまり render は自由入力ではなく、あらかじめ契約化されたキー集合にだけ作用する。

この前提があるからこそ、Split Deployment RAG でも chat と job と web に同じ基準で設定を配れる。単なるテンプレートコピーではなく、モジュールごとの base file を維持しながら profile 差分だけを当てる点が重要である。

Env Profile Rendering は contract と base template を結合し、生成済み `.env.` を経由して runtime process environment を準備する。

render スクリプトがしていること

スクリプトはまず contract を読み、module と profile の選択を正規化する。続いて base template を 1 行ずつ読み、環境変数代入に一致する行だけを override で置換する。ここで override 側に base template に存在しないキーがあるとエラーになるため、profile 定義がテンプレートから逸脱した時点で検出できる。

さらに --check モードでは生成せず、既存の .env.<profile> が render 結果と一致するかだけを見る。これは CI 上で drift を見つけるための用途に向いており、Check-Mode Drift Detection として切り出せる論点である。

モジュール差分の扱い

chat と job は Python サービスであり、search、OpenAI、Storage、webhook secret など多くの key を持つ一方、web の template はほぼ NEXT_PUBLIC_API_BASE_URL と NEXT_PUBLIC_APP_ENV に限られる。render スクリプトはこの差を抽象化しない。むしろ module ごとに base template を分けたまま、共通アルゴリズムだけを提供する。

この設計は、各 module の責務が異なることを隠さないという意味で妥当である。Module Env Catalogs を見ると、chat と job の env 空間が大きく異なり、それでも同じ render フローに乗っていることが分かる。

運用上の意味

Project Diana の README と runbook では、ローカル起動前に python tools/render_env_profiles.py --module ... --profile local を実行する前提になっている。これは runtime コードが load_dotenv() に依存せず、生成済み env file を process environment に流し込む構造だからである。つまり render は開発補助ではなく、起動手順の正式な一部である。

また、Policy-Enforced Retrieval で使う policy 系 env や top-k 系 env も、この render によって profile 別に供給される。したがって env profile rendering は、単なる設定配布ではなく挙動切替の入口でもある。

次に読むべき周辺

この仕組みをさらに分解するなら、contract と template の整合を確かめる Env Consistency Checks、module ごとの定義差分を追う Module Env Catalogs、生成されたファイルそのものを扱う Rendered Runtime Env Files が自然である。ローカル運用寄りには Local Docker Compose Profile もつながる。
