Taxonomy-Driven Metadata Validation

最終更新: 2026-03-24 19:00 JST
Taxonomy-Driven Metadata Validation は、モデルが提案するメタデータヒントを定義済みのタクソノミー（分類体系）に照合し、受理・拒否を判断する仕組みである。validated optional hint を taxonomy と metadata schema で正規化することで、検索フィルターの安全性と一貫性を保証する。

Taxonomy-Driven Metadata Validation は、Policy-Enforced Retrieval における validated optional filter の品質を保つための仕組みである。AIモデルが検索の絞り込みに使うメタデータヒント（ラベル、カテゴリ、インテントなど）を、あらかじめ定義されたタクソノミー（分類体系）に照合して受理・拒否を判断する。タクソノミーに存在しない値や、スキーマに適合しない形式のヒントは拒否されるため、不正確な絞り込みによる検索品質の低下を防ぐことができる。この検証は Retrieval Round Policy の optional_narrowing ラウンドで特に重要な役割を果たす。

タクソノミーの構造と役割

タクソノミーとは、メタデータの取りうる値を階層的に定義した分類体系である。Project Diana Agentic RAG では、ドキュメントの取り込み時に付与されるメタデータの種類ごとにタクソノミーが用意されている。主な分類軸は以下の通りである。

  • doc_categories: ドキュメントの種別（マニュアル、FAQ、技術仕様など）
  • task_intent: ユーザーの意図（質問、手順確認、トラブルシューティングなど）
  • chunk_keywords: チャンクに付与されたキーワード群
  • domain_labels: ドメイン固有のラベル（製品名、部門名など）

これらのタクソノミーは ドキュメント Ingest Pipeline の metadata ステージで参照され、取り込み時にもメタデータの正規化に使われる。検索時と取り込み時で同じタクソノミーを参照することで、メタデータの一貫性が保たれる。

バリデーションフロー

モデルが提案したメタデータヒントは、以下の手順で検証される。

  1. スキーマ検証: ヒントの形式（型、フィールド名、値の長さなど）が metadata schema に適合するかを確認する
  2. タクソノミー照合: 各フィールドの値がタクソノミーに登録されているかを確認する
  3. 正規化: 表記ゆれ（大文字小文字、別名、省略形など）をタクソノミーの正規形に変換する
  4. 受理・拒否の判定: 検証を通過したヒントのみが validated optional filter として採用される

拒否されたヒントは無視されるだけであり、検索自体が失敗するわけではない。Policy-Enforced Retrieval の中心不変条件により、mandatory filter は常に適用されるため、ヒントが全て拒否されても安全な検索結果が返る。

モデルが提案したヒントは Schema 検証とTaxonomy 照合を通過したものだけが validated optional filter として採用される。拒否されたヒントはログに記録されるが検索は継続する。

正規化と表記ゆれの吸収

タクソノミー照合の重要な機能の一つが正規化（ノーマライゼーション）である。モデルが生成するヒントは、タクソノミーに登録された正規形と完全に一致するとは限らない。たとえば以下のような表記ゆれが発生する。

  • 大文字・小文字の違い: "FAQ" と "faq"
  • 別名・省略形: "troubleshooting" と "TS"
  • 表現の揺れ: "ユーザーガイド" と "ユーザガイド"

正規化エンジンは、エイリアス（別名）マップとケースフォールディング（大文字小文字の統一）を組み合わせて、モデルの出力をタクソノミーの正規形に変換する。これにより、モデルが多少異なる表現を使っても検索フィルターとして正しく機能する。この仕組みは Docling Hybrid Chunking の metadata ステージでも共通して使われている。

エイリアス管理

エイリアスマップは、タクソノミーの各エントリに対して登録される別名の一覧である。新しいエイリアスの追加はタクソノミーの更新として管理され、環境変数コントラクト管理 と同様にバージョン管理される。エイリアスの衝突（複数の正規形に対して同じ別名が登録される状態）は、Taxonomy Conflict Resolution の仕組みで検出・解決される。

他コンポーネントとの連携

Taxonomy-Driven Metadata Validation は、検索パイプラインの複数のコンポーネントと連携して動作する。

Round Policy との連携

Retrieval Round Policy の optional_narrowing ラウンドでは、モデルが提案したヒントがこのバリデーションを通過したかどうかで挙動が変わる。有効なヒントが1つもなければ、narrowing ラウンドはスキップされ、optional_relaxed に進む。有効なヒントが存在すれば、それを使って検索結果を絞り込む。

Ingest Pipeline との整合

ドキュメント Ingest Pipeline の metadata ステージでは、同じタクソノミーを参照してドキュメントにメタデータを付与する。検索時のバリデーションと取り込み時のメタデータ付与が同一のタクソノミーを使うことで、「取り込み時に付与されたメタデータが検索時に使えない」という不整合を防いでいる。タクソノミーの更新時には、既存ドキュメントの再インデックスが必要になる場合があり、これは Taxonomy Migration Workflow で管理される。

トレースと観測可能性

バリデーションの結果（受理・拒否の判定とその理由）は、Workspace Trace UI で確認できる。どのヒントが拒否されたのか、なぜ拒否されたのか（スキーマ不適合、タクソノミー未登録など）がトレースに記録される。これにより、モデルのヒント精度の改善やタクソノミーの拡充に役立つフィードバックループが形成される。

タクソノミーのバージョン管理

タクソノミーは固定的なものではなく、業務の変化や新しいドキュメント種別の追加に応じて更新される。バージョン管理では以下の点が考慮されている。

  • 後方互換性: 新しいエントリの追加は常に安全だが、既存エントリの削除や名前変更は互換性を壊す可能性がある
  • 段階的な移行: エントリの廃止は deprecated フラグで管理し、一定期間は受理し続ける
  • CI での検証: タクソノミーの変更は Env Consistency Checks と同様に CI パイプラインで整合性を検証する

タクソノミーの変更履歴は Taxonomy Changelog Audit に記録され、いつ・誰が・なぜ変更したかを追跡できる。これは Live Retrieval Lanes における検索品質のデバッグにも役立つ。

設計の背景と判断

この仕組みが必要な理由は、AIモデルのメタデータ提案が常に正確とは限らないためである。モデルはプロンプトの文脈からヒントを生成するが、タクソノミーに存在しないカテゴリを「でっちあげる」ことがある（ハルシネーション）。チェックなしにこれを検索フィルターに使うと、検索結果がゼロ件になるか、意図しない絞り込みが起きる。

Policy-Enforced Retrieval の中心不変条件 effective_filter = mandatory_filter AND validated_optional_filter における "validated" の部分を、このタクソノミー駆動のバリデーションが担っている。つまり、モデルの提案がタクソノミーに照合されて初めて "validated" となり、effective filter に含められる。

将来的には、モデルのヒント精度に基づいて Hint Acceptance Rate Analytics を行い、タクソノミーの拡充やプロンプトの改善にフィードバックする仕組みが検討されている。
