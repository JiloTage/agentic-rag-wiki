Split Deployment RAG

最終更新: 2026-03-24 18:08 JST
Split Deployment RAG は、チャット・文書の取り込み・Web 画面を別々のサービスとして動かすやり方です。こうすると、チャットと文書の処理がお互いに影響しにくくなります。それぞれを別々に増やしたり直したりできるのがメリットです。

Split Deployment RAG は、RAG（質問に答えるために関連する文書を検索して使う仕組み）を、1つのプログラムにまとめずに別々のサービスとして動かすやり方です。Project Diana Agentic RAG では、apps/web（画面）、apps/chat（チャットAPI）、apps/job（文書の取り込み）、infra（インフラ）に分かれています。ドキュメント Ingest Pipeline や 環境変数コントラクト管理 も、この分け方を前提にしています。

なぜ分離するのか

「ユーザーにすぐ答えを返す処理」と「大きなファイルを読み取って検索データを作る処理」は、性質がまったく違います。Project Diana では、chat API がユーザーとの会話を担当し、ingest job（文書の取り込みを行う裏方）が時間のかかる処理を受け持ちます。こうしておくと、文書の取り込みが遅れても、チャットは止まりません。

もうひとつの良い点は、それぞれに必要な権限だけを与えられることです。API は job を起動する権限だけを持ちます。文書の読み取りは job が行います。Policy-Enforced Retrieval（検索にルールを適用する仕組み）は API 側にまとめて、重い処理は job に任せます。

Web 画面、API、取り込みジョブを別々に動かします。chat と job はそれぞれ Azure Search や Azure OpenAI を使います。

責務境界

Web UI の境界

apps/web は Next.js で作られた画面です。チャット、ファイル一覧、取り込み状況、トレース表示をユーザーに見せます。ここでは検索や OCR を直接実行せず、API を呼ぶだけに徹しています。そのため、画面の変更と取り込み処理の変更を、お互いに影響なく進められます。

Chat API の境界

apps/chat は、認証、スレッド管理、ファイル管理、チャット応答、ストリーミング、監視、取り込みジョブの起動といった窓口を担います。ここで大事なのは、API が自分で文書の変換をすべてやるのではなく、必要なときに job を起動する「まとめ役」として動くことです。つまり API はユーザーとの同期的なやりとりを守りつつ、裏の処理の入口になっています。

Ingest Job の境界

apps/job は、準備・解析・OCR・分割・メタデータ付与・インデックス登録・コールバック通知を順に行う非同期ワーカーです。CPU やメモリを多く使う重い処理をこの job 側に分けることで、chat API を軽く保てます。ここで作られるデータの詳細は ドキュメント Ingest Pipeline の記事で読めます。

配備にどう落ちるか

docker-compose.yml では、chat、web、job がそれぞれ別のサービスとして書かれています。chat と web にはヘルスチェック（正常に動いているかの確認）があり、web は chat が正常に起動してから立ち上がります。job は compose の profile を使って手動で起動します。つまり、常に動いているサービスと、必要なときだけ動かすサービスが分かれています。

infra/main.bicep では、この分離が Azure 上の実際のサービスになります。API は Container App、文書取り込みは手動で起動する Container Apps Job、Web 画面は Static Web App として定義されています。API には「job を起動する権限だけ」が与えられています。この構成の詳しい話は Azure Container Apps Topology や API-Job Trigger Boundary の記事で読めます。

ローカル検証フロー

サービスを分けて動かすときは、起動前に環境変数をどうそろえるかが大事です。Project Diana では tools/render_env_profiles.py というスクリプトが、各モジュールの .env.example と環境ごとの設定をまとめて .env.local を作ってくれます。chat・job・web それぞれに必要な値を配ってくれるのです。このように、split deployment はただコンテナを分けるだけでなく、環境変数コントラクト管理 とセットで使う仕組みです。

ローカルでは docker compose up --build で chat と web をまとめて起動します。文書を取り込みたいときは docker compose --profile job run --rm job で ingest を実行します。こうすることで、ユーザー向けの機能と文書の取り込みを、別々のタイミングで試せます。

設計上の含意

split deployment は、スケーリングや可用性の面では有利ですが、設定の同期・コールバックの取り決め・監視・デプロイ順序の管理が必要になります。Project Diana が runbook・env contract・段階デプロイ・trace UI を同じリポジトリに置いているのは、この管理コストに対応するためです。つまり、サービスを分けるだけでは不十分で、分けた後のつなぎ目をドキュメントとコードで維持していく必要があります。

次に読むと理解が深まる記事としては、環境変数コントラクト管理、ドキュメント Ingest Pipeline、Policy-Enforced Retrieval、Workspace Trace UI があります。デプロイの詳細については Local Docker Compose Profile も参考になります。
