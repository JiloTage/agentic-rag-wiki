Split Deployment RAG

最終更新: 2026-03-24 18:08 JST
Split Deployment RAG は、対話 API、文書 ingest、Web UI を別の実行単位として分離する RAG 構成である。Project Diana ではこの分離により、応答経路と indexing 経路の失敗境界、権限境界、スケール戦略を切り分けている。

Split Deployment RAG とは、RAG システムの主要責務を単一プロセスに押し込めず、別サービスとして運用する考え方である。Project Diana Agentic RAG では apps/web、apps/chat、apps/job、infra がこの分離を担い、ドキュメント Ingest Pipeline や 環境変数コントラクト管理 もその前提の上に組み立てられている。

なぜ分離するのか

RAG の実運用では、ユーザーに即時応答を返す経路と、大きなファイルを解析して検索インデックスを更新する経路では、必要な計算資源も失敗の性質も異なる。Project Diana の README と local runbook はこの違いを前提にしており、chat API は request/response の境界を守り、ingest job は時間のかかる解析処理を非同期実行する。これにより ingest の遅延や OCR 失敗が、そのまま chat サービスの可用性低下に直結しにくくなる。

分離のもう一つの利点は権限の局所化である。API 側は job を起動する最小限の権限だけを持ち、文書解析そのものは job 側で完結する。この構造は Policy-Enforced Retrieval を API 側に集中させつつ、重い文書処理を別の失敗境界へ逃がすためのものだと言える。

Split Deployment RAG では、Web UI、API、ingest job を分離し、chat と job が Azure Search / Azure OpenAI を別経路で利用する。

責務境界

Web UI の境界

apps/web は Next.js ベースの workspace であり、チャット、ファイル一覧、ingest 状態、trace 表示をユーザーへ見せる。ここは体験面の境界であり、検索や OCR を直接実行せず、API を呼ぶ側に徹する。結果として UI のビルドや画面遷移の変更は、ingest worker の依存関係と独立して進めやすい。

Chat API の境界

apps/chat は認証、threads、files、chat、streaming、observability、ingest 起動の窓口を担う。ここで重要なのは、API 自身がすべての文書変換を行うのではなく、必要に応じて job を起動する orchestration layer として振る舞う点である。つまり API はユーザーとの同期境界を守りつつ、バックグラウンド処理の入口になる。

Ingest Job の境界

apps/job は prepare、parse、OCR、chunk、metadata、index、callback を流す非同期 worker である。CPU、メモリ、外部サービス依存が重くなりやすい処理をこの job 側へ隔離することで、chat API のプロセスを軽く保ちやすい。ここで作られる成果物は ドキュメント Ingest Pipeline の記事でさらに細かく追える。

配備にどう落ちるか

docker-compose.yml は local-first で、chat、web、job を個別サービスとして定義する。chat と web には health check があり、web は chat の healthy 状態に依存して起動する。job は compose の profile で明示起動されるため、常時動かすべき経路と必要時だけ動かす経路が分かれている。

infra/main.bicep ではこの分離が Azure 上の実体になる。API は Container App、ingest は manual trigger の Container Apps Job、web は Static Web App として宣言され、API に対しては job スコープに限定した起動権限だけが与えられる。これは Azure Container Apps Topology や API-Job Trigger Boundary といった派生テーマにそのままつながる。

ローカル検証フロー

split deployment では、起動前に env をどうそろえるかが重要になる。Project Diana では tools/render_env_profiles.py が各モジュールの .env.example と profile override を束ねて .env.local を生成し、chat/job/web のそれぞれに必要な値を分配する。このため split deployment は単なるコンテナ分割ではなく、環境変数コントラクト管理 と一体で運用されている。

ローカルでは docker compose up --build によって chat と web をまとめて立ち上げ、必要に応じて docker compose --profile job run --rm job で ingest を流す。これによりユーザー向け操作と文書投入の検証を別テンポで回せる。

設計上の含意

split deployment は、スケールと可用性の面では有利だが、設定同期、コールバック契約、観測性、ロールアウト順序の管理を強く要求する。Project Diana が runbook、env contract、phase rollout、trace UI を同じ repo に置いているのは、このコストを補うためである。言い換えれば、単にサービスを分けるだけでは不十分で、分けた後の接着面をドキュメントとコードで維持する必要がある。

そのため、次に読むべき記事としては 環境変数コントラクト管理、ドキュメント Ingest Pipeline、Policy-Enforced Retrieval、Workspace Trace UI が自然である。さらに配備実装の細部として Local Docker Compose Profile を切り出す余地もある。
