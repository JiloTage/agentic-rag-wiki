Split Deployment RAG

最終更新: 2026-03-24 18:08 JST
Split Deployment RAG は、チャット・文書の取り込み・Web 画面を別々のサービスとして動かすやり方です。こうすると、チャットと文書の処理がお互いに影響しにくくなります。それぞれを別々に増やしたり直したりできるのがメリットです。

Split Deployment RAG は、RAG（質問に答えるために関連する文書を検索して使う仕組み）を、1つのプログラムにまとめずに別々のサービスとして動かすやり方です。Project Diana Agentic RAG では、apps/web（画面）、apps/chat（チャットAPI）、apps/job（文書の取り込み）、infra（インフラ）に分かれています。ドキュメント Ingest Pipeline や 環境変数コントラクト管理 も、この分け方を前提にしています。

なぜ分離するのか

「ユーザーにすぐ答えを返す処理」と「大きなファイルを読み取って検索データを作る処理」は、性質がまったく違います。Project Diana では、chat API がユーザーとの会話を担当し、ingest job（文書の取り込みを行う裏方）が時間のかかる処理を受け持ちます。こうしておくと、文書の取り込みが遅れても、チャットは止まりません。

もうひとつの良い点は、それぞれに必要な権限だけを与えられることです。API は job を起動する権限だけを持ちます。文書の読み取りは job が行います。Policy-Enforced Retrieval（検索にルールを適用する仕組み）は API 側にまとめて、重い処理は job に任せます。

Web 画面、API、取り込みジョブを別々に動かします。chat と job はそれぞれ Azure Search や Azure OpenAI を使います。

責務境界

Web UI の境界

apps/web は Next.js で作った画面です。チャット画面、ファイルの一覧、取り込みの進み具合、処理の記録などを表示します。この画面は検索や OCR（画像から文字を読み取ること）を自分では行いません。API を呼ぶだけです。そのおかげで、画面を変えるときに取り込み処理を気にしなくて済みます。

Chat API の境界

apps/chat は、ログイン確認、スレッド管理、ファイル管理、チャットの返答、データの送信、監視、取り込み job の起動などを行います。大事なのは、API が自分で文書を変換するのではなく、必要なときに job を動かす「まとめ役」だということです。ユーザーとのやりとりを止めずに、裏側の処理を始められます。

Ingest Job の境界

apps/job は、文書の準備、読み取り、OCR、分割、情報の付与、検索データへの登録、結果の通知を順番に行います。CPU やメモリをたくさん使う重い処理をこちらに分けることで、chat API を軽くできます。くわしくは ドキュメント Ingest Pipeline をご覧ください。

配備にどう落ちるか

docker-compose.yml では、chat、web、job が別々のサービスとして書かれています。chat と web にはヘルスチェック（ちゃんと動いているかの確認）があります。web は chat が起動してから立ち上がります。job は必要なときだけ手動で起動します。

infra/main.bicep では、Azure 上のサービスとしてデプロイされます。API は Container App、文書取り込みは Container Apps Job、画面は Static Web App です。API には「job を起動する権限だけ」が与えられます。くわしくは Azure Container Apps Topology や API-Job Trigger Boundary をご覧ください。

ローカル検証フロー

サービスを分けて動かすときは、環境変数（アプリの設定値）をそろえることが大切です。tools/render_env_profiles.py というスクリプトが、各モジュールの .env.example と環境ごとの設定をまとめて .env.local を作ってくれます。chat・job・web にそれぞれ必要な値を配ってくれます。このように、サービスを分けるだけでなく、環境変数コントラクト管理 とセットで使います。

ローカルでは docker compose up --build で chat と web を起動します。文書を取り込みたいときは docker compose --profile job run --rm job を使います。こうすると、チャットと文書取り込みを別々に試せます。

設計上の含意

サービスを分けると、増やしたいものだけ増やせるし、壊れにくくなります。ただし、設定をそろえること、結果の通知のやり方、監視、デプロイの順番などを管理する必要があります。Project Diana では runbook、env contract、段階デプロイ、trace UI を同じリポジトリに置いて、この管理をしやすくしています。
