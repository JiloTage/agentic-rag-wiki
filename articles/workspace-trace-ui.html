Workspace Trace UI

最終更新: 2026-03-24 18:45 JST
Workspace Trace UI は、Project Diana の Next.js ワークスペース画面に組み込まれたトレース表示機能である。retrieval round の検証ログ、ingest job の進捗、policy enforcement の判断過程をリアルタイムで可視化し、説明責任と運用デバッグを支える。

Workspace Trace UI は、Project Diana Agentic RAG の apps/web に実装された、説明責任（アカウンタビリティ）と運用デバッグを担う UI コンポーネント群である。Next.js ワークスペース UI の一部として、TraceModal と WorkspaceShell を中心に構成され、Retrieval Round Policy の検証ログ、ドキュメント Ingest Pipeline の進捗状況、Policy-Enforced Retrieval の判断過程をユーザーが直接確認できるようにする。

Trace UI の役割

RAG システムでは「AI がなぜその回答をしたのか」が見えにくいという課題がある。Workspace Trace UI は、この課題に正面から取り組む機能である。具体的には次の 3 つの情報を表示する。

  • Retrieval ログ: 各 retrieval round（検索の繰り返し）でどのフィルタが適用され、何件のチャンクが返されたか。Retrieval Round Policy の mandatory_broad、optional_narrowing、optional_relaxed の各フェーズを時系列で追跡できる。
  • Ingest 進捗: ドキュメント Ingest Pipeline の prepare、parse、Picture OCR、chunk、metadata、index の各ステージがどこまで進んだか。エラーが起きたステージも赤く表示される。
  • Policy 判断過程: Policy-Enforced Retrieval で mandatory filter（必須フィルタ）がどう構成され、モデルからの optional hint（任意の絞り込み提案）がどう検証されたかの詳細。

これらの情報は、開発者のデバッグだけでなく、管理者が「検索が正しく動いているか」を確認するための監査証跡（オーディットトレイル）としても機能する。

主要コンポーネント

TraceModal

TraceModal は、チャット応答に紐づくトレース情報をモーダルウィンドウ（画面の前面に表示される小窓）で表示するコンポーネントである。ユーザーがチャットの回答横にある「トレース」ボタンをクリックすると開く。内部では retrieval round ごとのフィルタ条件、ヒット件数、使用されたチャンクの一覧、policy の判定結果がタイムライン形式で並ぶ。

TraceModal が表示するデータは、apps/chat の observability エンドポイントから取得される。Split Deployment RAG の構成では API サーバーが検索処理を担当するため、トレースデータも API 側で生成され、Web UI はそれを読み取って描画するだけという役割分担になっている。

WorkspaceShell

WorkspaceShell は、ワークスペース全体のレイアウトを管理する大枠コンポーネントである。ChatPanel、FilesPanel、IngestPanel、ThreadPanel、TraceModal をタブやパネルとして束ね、ユーザーがシームレスに切り替えられるようにする。WorkspaceShell のサイドバーには ingest job のリアルタイムステータスバーが常時表示されており、バックグラウンドで動いている文書取り込みの進捗を把握できる。

IngestPanel

IngestPanel は、ドキュメント Ingest Pipeline のジョブ一覧と詳細を表示する専用パネルである。各ジョブのステージ進捗をプログレスバーで示し、失敗したステージにはエラーコードと失敗理由が添えられる。Picture OCR Stage のように部分成功が許容されるステージでは、成功数と失敗数の内訳も確認できる。

Workspace Trace UI のコンポーネント構成図。WorkspaceShell が ChatPanel、TraceModal、IngestPanel を束ね、それぞれが API サーバーや Job ワーカーからトレースデータを取得する。

トレースデータの流れ

トレースデータは、検索処理の実行時にリアルタイムで生成される。具体的な流れは次のとおりである。

  1. ユーザーがチャットで質問を送信すると、apps/chat の chat エンドポイントが Retrieval Round Policy に従って検索を開始する。
  2. 各 retrieval round の開始時に、適用されるフィルタ（mandatory filter と accepted hint）、Optimization Priority Modes に基づく予算配分、検索対象のインデックスがログに記録される。
  3. round 完了時に、ヒット件数、スコア分布、選択されたチャンク ID、消費された予算がログに追加される。
  4. 全 round 終了後、Policy-Enforced Retrieval の最終判定（どの hint が採用され、どれが拒否されたか）がまとめられる。
  5. これらのログは observability エンドポイント経由で TraceModal に渡される。

ingest のトレースデータは別の経路をたどる。apps/job がステージごとの進捗を callback（コールバック）で apps/chat に通知し、API サーバーが Azure Container Apps の Storage Table に状態を保存する。IngestPanel はこのテーブルをポーリング（定期的に問い合わせ）して画面を更新する。

Retrieval Trace の詳細表示

TraceModal の中でもっとも重要なのは、retrieval trace の詳細表示である。1 回のチャット応答に対して、複数の retrieval round が実行されることがある。各 round は次の情報を持つ。

  • Round 種別: mandatory_broad（必須の広い検索）、optional_narrowing（任意の絞り込み検索）、optional_relaxed（任意の緩和検索）のいずれか。
  • フィルタ構成: mandatory filter のキーと値、モデルが提案した hint のうち採用されたもの、拒否されたもの。
  • 検索結果サマリ: ヒット件数、上位チャンクのスコア、引用に使われたチャンクのハイライト。
  • 予算消費: この round で使われた検索予算と、残りの予算。Optimization Priority Modes が quality モードか latency モードかによって予算の総量が変わる。

この情報があれば、「なぜこの文書が引用されたのか」「なぜこの文書は見つからなかったのか」を開発者が追跡できる。たとえば、mandatory filter が厳しすぎてヒット 0 件になった場合、round の種別が optional_relaxed に進んだことがトレースからわかる。

Hint Validation ログ

モデルが提案する metadata hint（メタデータによる検索の絞り込み提案）は、Policy-Enforced Retrieval のルールに従って検証される。TraceModal では、各 hint について「提案値」「検証結果（accept / reject）」「拒否理由」が表形式で表示される。これにより、モデルが不適切な hint を出していないか、あるいは有用な hint が誤って拒否されていないかを確認できる。

Ingest Trace の詳細表示

IngestPanel は、ドキュメント Ingest Pipeline の各ジョブをカード形式で一覧表示する。1 つのジョブカードには次の情報が含まれる。

  • ファイル名とジョブ ID: どの文書の取り込みかを識別する。
  • ステージ進捗バー: prepare、parse、picture_ocr、chunk、metadata、index の 6 ステージを色分けで表示。完了は緑、進行中は青、失敗は赤、待機中はグレー。
  • エラー詳細: 失敗ステージをクリックすると、エラーコードとスタックトレースの要約が展開される。
  • 部分成功の内訳: Picture OCR Stage のように複数の画像を処理するステージでは、「10 枚中 8 枚成功、2 枚失敗」のような内訳が表示される。

この画面があることで、文書取り込みが途中で止まったときに「どのステージで何が起きたか」を即座に特定でき、Split Deployment RAG の構成で分離されている job サービスのトラブルシューティングが効率化される。

設計上の方針

Workspace Trace UI の設計には、いくつかの明確な方針がある。

読み取り専用のトレース

TraceModal は情報の表示に特化しており、トレースデータを編集したり、検索のやり直しを指示したりする機能は持たない。これは意図的な設計で、トレースが「事実の記録」であることを保証する。操作ログと監査証跡を同じデータソースから生成できるようにすることで、説明責任を果たしやすくなる。

段階的な情報開示

最初は round の種別とヒット件数だけを表示し、クリックで詳細を展開する。すべての情報を一度に表示すると、日常的な使い方では情報過多になるためである。開発者向けの詳細（フィルタの具体的な値、チャンク ID、スコアの生値）は 2 段階目以降に配置される。

API 駆動のデータ取得

Web UI はトレースデータの生成に一切関与しない。すべてのトレースデータは apps/chat の API から取得される。この方針は Split Deployment RAG の責務分離と一致しており、Web UI を差し替えても同じトレースデータを利用できる。将来的に Trace Export API を追加すれば、外部の監視ツールからもアクセス可能になる。

運用での活用シーン

Workspace Trace UI は、次のような場面で活用される。

  • 検索精度の検証: 新しい文書を取り込んだ後に、期待するチャンクが retrieval で返されるかを確認する。Optimization Priority Modes を切り替えた際の影響確認にも使える。
  • Ingest エラーの調査: PDF の取り込みが失敗した場合、IngestPanel で parse ステージの失敗を確認し、Docling Hybrid Chunking のログと突き合わせる。
  • Policy 設定の調整: mandatory filter が厳しすぎて検索結果が返らないケースを TraceModal で発見し、Retrieval Round Policy のパラメータを調整する根拠にする。
  • 監査対応: 「なぜこの回答がされたか」を第三者に説明する必要がある場合、TraceModal のスクリーンショットやデータエクスポートを証拠として使う。
  • パフォーマンス分析: 各 round の応答時間を確認し、Trace Latency Breakdown の分析に活用する。

今後の拡張としては、トレースデータの 外部エクスポート機能、2 つのトレースの差分比較、異常検知アラート などが検討されている。これらの機能が実現すれば、Workspace Trace UI は単なるデバッグツールから、RAG の品質管理基盤へと発展する。
