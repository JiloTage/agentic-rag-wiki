ドキュメント Ingest Pipeline

最終更新: 2026-03-24 18:08 JST
ドキュメント Ingest Pipeline は、アップロードされたファイルを「あとから検索できるデータ」に変える処理の流れです。ファイルの確認から結果の通知まで、いくつかのステップに分けて順番に実行します。

ドキュメント Ingest Pipeline は、ユーザーがアップロードしたファイルを、あとから検索できるかたちに変える処理です。Project Diana Agentic RAG では、apps/job/job_src/entrypoint.py がこの処理の中心です。Split Deployment RAG（サービスを分けて動かすやり方）では、裏方の job として動きます。

パイプラインの全体像

処理は validate（確認）、prepare（準備）、parse（読み取り）、picture OCR（画像の文字認識）、chunk（小さく分ける）、metadata（情報の付与）、index（検索データに登録）、callback（結果の通知）の順に進みます。chunk と metadata は同時に実行されます。この処理のゴールは、ただファイルを読むことではなく、「あとから見つけやすいデータ」を作ることです。

ingest job は確認から通知まで順番に進みます。chunk と metadata だけは同時に動きます。

Prepare stage

job_src.prepare は、ファイルをダウンロードして、形式を調べて、古い Office ファイルを新しい形式に変えます。ファイルが大きすぎないか、パスワードがかかっていないか、ページ数が多すぎないかもここで確認します。問題があるファイルは、この段階で止めます。次の処理に壊れたデータを渡さないための「門番」です。

古い Office ファイルは LibreOffice で新しい形式に変えます。うまくいかないときは PDF に変えます。エラーには prepare_unsupported_format や prepare_page_limit_exceeded のような名前がつくので、あとから原因を調べやすいです。

Parse と OCR fallback

job_src.parsing は、Docling というツールを使って文書の中身を読み取ります。PDF や画像のときは、Azure Document Intelligence の OCR（画像から文字を読み取る機能）も使えます。parse_mode=auto にすると、まず速い方法で読み取ります。うまく読めなかったときだけ OCR を使います。こうすると、お金と時間を節約できます。

速い方法で本文が読めても、画像の中に文字がある場合は、その部分だけ OCR を行います。「速い方法か OCR か」の二択ではなく、必要なところだけ OCR を使う仕組みです。くわしくは Picture OCR Stage や Parse Mode Fallback をご覧ください。

Chunk・Metadata・Index

Chunk stage

job_src.chunking は、Docling HybridChunker を使って、文書を小さなかたまり（chunk）に分けます。見出しや段落のまとまりを考えながら分けます。できたデータには、検索用の要約 search_text、前後の文脈 context_full、見出し、キーワード、画像から読んだ文字なども入っています。こうしておくと、あとから検索したときに見つけやすくなります。

Metadata stage

metadata（メタデータ）の処理では、本文と画像の文字をまとめて分析して、カテゴリ、キーワード、文書の目的などを取り出します。この処理が失敗しても、警告を出すだけで全体は止めません。メタデータが取れなければ、空のデータを使います。「あると便利だけど必須ではない」処理なので、このようにしています。

Index stage

最後に、情報を付けた chunk を Azure Search（検索サービス）に書き込みます。元のファイルをそのまま保存するのではなく、検索しやすい形に変えて登録します。この分け方が大事なので、Docling Hybrid Chunking もあわせてご覧ください。

Callback と failure model

ingest job は、うまくいっても失敗しても、結果を callback（結果通知）として API に返します。失敗したときは error_code（エラーの種類）とくわしい情報がつきます。どのステップで失敗したかもわかります。API 側はこの情報をもとに、ユーザーに「処理中」「完了」「失敗」と表示できます。

この callback の仕組みは、サービスを分けて動かすときにとても大事です。job と API はお互いの中身を知らなくてよいので、結果の通知だけでやりとりします。UI 側では処理の履歴が表示されるので、Workspace Trace UI もあわせて見ると流れがわかりやすくなります。
