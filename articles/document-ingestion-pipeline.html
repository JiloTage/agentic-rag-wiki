ドキュメント Ingest Pipeline

最終更新: 2026-03-24 18:08 JST
ドキュメント Ingest Pipeline は、validate から callback までを段階分割した非同期処理系である。Project Diana の job サービスでは prepare、parse、picture OCR、chunk、metadata、index を明示的に分離し、失敗コードと callback で API 境界に返している。

ドキュメント Ingest Pipeline は、アップロード済み文書を検索可能な知識へ変換する一連の処理である。Project Diana Agentic RAG では apps/job/job_src/entrypoint.py が pipeline の骨格を持ち、Split Deployment RAG の worker 側責務として切り出されている。

パイプラインの全体像

entrypoint の主経路は validate、prepare、parse、picture OCR、chunk、metadata、index、callback の順に進む。特に chunk と metadata は並列に実行され、最後に index 書き込みと callback 通知へ合流する。これは ingest を単なる PDF 解析ではなく、構造化済み検索素材を作る処理と見なしていることを示している。

ingest job は validate から callback までを段階化し、chunk と metadata を並列に流す。

Prepare stage

job_src.prepare はダウンロード、拡張子ベースの format 判定、legacy Office 変換、事前 guard を担当する。サイズ制限、暗号化 PDF / Office 検知、ページ数上限、blob URL policy がここで先に評価され、以降の Docling 処理へ壊れた入力を持ち込まない。つまり prepare は parser の前段にある保安層である。

legacy Office 形式は LibreOffice headless で近代形式へ変換し、失敗時は PDF への fallback も持つ。ここでのエラーは prepare_unsupported_format や prepare_page_limit_exceeded などのコードで分類され、後段が generic 例外に埋もれないようになっている。

Parse と OCR fallback

job_src.parsing は Docling の converter を組み立て、PDF / 画像では Azure Document Intelligence を使う OCR plugin を選択できる。parse_mode=auto の場合はまず fast parse を試し、テキスト品質が閾値を満たさないときだけ OCR へ落ちる。これはコストと速度を抑えながら、必要なときだけ精度を取りに行く戦略である。

さらに fast parse が十分な場合でも、画像内テキスト抽出に必要なら OCR snapshot だけを追加で取得する。つまり parse は単純な二択ではなく、本文品質と picture OCR の双方を見ながら最小限の OCR を行う。ここは Picture OCR Stage や Parse Mode Fallback として切り出せる。

Chunk・Metadata・Index

Chunk stage

job_src.chunking は Docling HybridChunker を使い、構造を考慮した chunk を作る。出力は生テキストだけではなく、検索用要約 search_text、拡張文脈 context_full、見出しチェーン、chunk keywords、picture OCR 由来の局所テキストまで含む。これは検索インデックスを「単なる分割結果」ではなく、後段の retrieval に最適化された素材として作ることを意味する。

Metadata stage

metadata 生成では本文と picture OCR の内容をまとめた analysis input を使い、カテゴリやキーワードや task intent を抽出する。失敗時は degrade 可能なエラーを warning として扱い、document metadata を空配列で返して ingest 全体を止めない。これは解析補助と必須パスを分ける設計である。

Index stage

最後に enriched chunk 群が Azure Search へ書き込まれる。つまり ingest pipeline の本質は「元文書を保存すること」ではなく、「retrieval が使いやすい形へ再構成すること」にある。ゆえに Docling Hybrid Chunking はこの pipeline の核心候補になる。

Callback と failure model

ingest job は成功でも失敗でも callback payload を準備し、API 側へ結果を返す。失敗時には error_code と detail が付与され、stage も記録される。これにより API は非同期処理の詳細を直接抱え込まずに済み、ユーザー向けには状態遷移だけを表示しつつ、運用者は詳細コードを追える。

この callback 契約は split deployment の要所であり、job と API が疎結合であるための条件でもある。さらに UI 側では trace や ingest panel へつながるため、Workspace Trace UI とも接続して理解すると全体像が見えやすい。
