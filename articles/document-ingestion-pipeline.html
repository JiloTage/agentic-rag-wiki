ドキュメント Ingest Pipeline

最終更新: 2026-03-24 18:08 JST
ドキュメント Ingest Pipeline（文書取り込みパイプライン）は、アップロードされた文書を検索できるデータに変える処理の流れです。validate（検証）から callback（結果通知）まで、いくつかのステップに分かれています。Project Diana の job サービスでは、prepare、parse、picture OCR、chunk、metadata、index の各ステップを分けて実行し、結果や失敗の情報を API に返します。

ドキュメント Ingest Pipeline は、アップロードされた文書を、あとから検索できるデータに変える処理のことです。Project Diana Agentic RAG では、apps/job/job_src/entrypoint.py がこの処理の中心になっています。Split Deployment RAG（サービスを分けて動かす構成）では、worker（裏方の処理役）の仕事として切り出されています。

パイプラインの全体像

処理は validate、prepare、parse、picture OCR、chunk、metadata、index、callback の順に進みます。chunk（文書を小さく区切る処理）と metadata（分類やキーワードの抽出）は同時に実行されます。最後に index（検索インデックスへの書き込み）と callback（結果の通知）が行われます。つまり、この処理は単に PDF を読むだけではなく、「あとから検索しやすいデータ」を作ることが目的です。

ingest job は validate から callback まで順番に処理を進めます。chunk と metadata は同時に実行されます。

Prepare stage

job_src.prepare は、ファイルのダウンロード、ファイル形式の判定、古い Office 形式の変換、事前チェックを行います。ファイルサイズの上限チェック、暗号化された PDF や Office ファイルの検出、ページ数の上限チェック、blob URL のルール確認がここで行われます。問題のあるファイルは、この段階で止められます。つまり prepare は、次の処理に壊れたデータを渡さないための「門番」です。

古い Office 形式のファイルは、LibreOffice を使って新しい形式に変換します。変換できなかった場合は PDF に変換しようとします。ここで起きたエラーには prepare_unsupported_format や prepare_page_limit_exceeded などの名前がつけられるので、あとから原因を調べやすくなっています。

Parse と OCR fallback

job_src.parsing は、Docling というライブラリを使って文書の中身を読み取ります。PDF や画像の場合は、Azure Document Intelligence の OCR（画像からテキストを読み取る機能）を使うこともできます。parse_mode=auto に設定すると、まず高速な方法で読み取りを試みます。テキストの品質が十分でないときだけ、OCR を使います。こうすることで、コストと時間をおさえながら、必要なときだけ精度の高い読み取りができます。

高速な読み取りで本文が十分に取れた場合でも、画像の中にあるテキストが必要なら、画像部分だけ OCR を追加で行います。つまり parse は「高速か OCR か」の二択ではなく、本文と画像の両方を見ながら、最小限の OCR で済ませる仕組みです。詳しくは Picture OCR Stage や Parse Mode Fallback をご覧ください。

Chunk・Metadata・Index

Chunk stage

job_src.chunking は、Docling HybridChunker を使って、文書の構造（見出しや段落など）を考えながら小さなかたまり（chunk）に分けます。出力にはただのテキストだけでなく、検索用の要約 search_text、前後の文脈を含む context_full、見出しの階層、キーワード、画像から読み取ったテキストなども含まれます。こうすることで、あとから検索するときに見つけやすいデータになります。

Metadata stage

metadata の生成では、本文と画像から読み取ったテキストをまとめて分析し、カテゴリ、キーワード、文書の目的（task intent）などを取り出します。もしこの処理が失敗しても、警告を出すだけで全体の処理は止めません。metadata が取れなかった場合は空のデータを返します。こうすることで、「あると便利だが必須ではない処理」と「絶対に必要な処理」を分けています。

Index stage

最後に、情報を追加した chunk を Azure Search（検索サービス）に書き込みます。つまり、この処理の目的は「元の文書をそのまま保存すること」ではなく、「あとから検索しやすい形に作り変えること」です。そのため、Docling Hybrid Chunking（構造を考えた分割のやり方）がこの処理の中で特に重要な部分です。

Callback と failure model

ingest job は、成功しても失敗しても、結果を callback（結果通知）として API に返します。失敗したときには error_code（エラーの種類）と詳細情報がつき、どのステップで失敗したかも記録されます。こうすると、API 側は処理の細かい内容を知らなくても、ユーザーに「処理中」「完了」「失敗」などの状態だけを見せられます。運用する人は、エラーコードを見て原因を調べられます。

この callback の仕組みは、サービスを分けて動かす構成で特に大事です。job と API がゆるくつながっている（お互いの中身を知らなくてよい）ための条件でもあります。また UI 側では、処理の履歴や取り込み画面に表示されるため、Workspace Trace UI もあわせて見ると全体の流れがわかりやすくなります。
