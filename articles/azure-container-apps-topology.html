Azure Container Apps Topology

最終更新: 2026-03-24 18:08 JST
Azure Container Apps Topology は、Project Diana の infra/main.bicep が split deployment を Azure 上の API Container App、manual Container Apps Job、Static Web App、Storage Table 群として具体化する配備構成である。API には ingest job だけを起動する scoped role assignment が与えられる。

Azure Container Apps Topology は、Split Deployment RAG を Azure の実リソースへ落とした構成である。Project Diana Agentic RAG の infra/main.bicep は、API を Container App、ingest を manual trigger の Container Apps Job、UI を Static Web App として分離し、単一プロセス化ではなく役割別配備を明示している。

リソース配置

Bicep はまず managed environment、image、search / OpenAI endpoint、secret 群を受け取り、diana-api-<env>、diana-ingest-<env>、diana-web-<env> を組み立てる。さらに storage account を作成し、ingeststatus と chathistory の table を切るため、API と job の責務分離だけでなく状態格納先も topology の一部として定義される。

API、manual ingest job、Static Web App、storage tables を同一テンプレートで束ね、job 起動だけを API に委譲する。

API Container App

API app は external ingress と targetPort: 8000 を持ち、1Gi / 0.5 CPU の単位で minReplicas: 1、maxReplicas: 3 を取る。secret には Search key、OpenAI key、JWT secret、ingest webhook secret、auth user JSON が入り、runtime env には AZURE_CONTAINERAPPS_JOB_NAME や table 名も注入される。つまり Policy-Enforced Retrieval と ingest 起動 API は、同じ API app の runtime contract 上に載っている。

Manual ingest job

ingest 側は Microsoft.App/jobs として manual trigger で定義され、replicaTimeout: 900、replicaRetryLimit: 1、parallelism: 1 を持つ。job container は 2Gi / 1 CPU と API より重く、ドキュメント Ingest Pipeline を想定した資源配分になっている。検索や OpenAI の secret は共有するが、JWT secret など API 専用の責務は持たない。

権限の絞り込み

Bicep は API app の system-assigned identity に対し、subscription ではなく ingest job resource だけを scope とする role assignment を付ける。コメントにもある通り built-in role は Contributor だが、割り当て範囲を job に限定することで、API が持つ権限を「ingest start に必要な最小セット」へ押し込めている。これは API-Job Trigger Boundary や Job Scope Role Assignment にそのまま接続する。

ローカルとの対応

docker-compose.yml は cloud topology のローカル縮約版である。chat と web は常時サービスとして立ち上がり、web は chat の healthy 状態に依存する。job は profiles: [\"job\"] に入っており、必要時だけ明示起動される。この対比によって Local Docker Compose Profile と Azure 配備の mental model が一致する。

ただし local と cloud の差分は env 配布で吸収しなければならない。したがって topology の再現性は Env Profile Rendering と Env Consistency Checks に大きく依存する。

運用上の含意

この topology では API と job が storage table を共有しつつ、UI は Static Web App として別配備になる。そのため rollout の論点は、コードだけでなく status table schema、secret wiring、build output path にも及ぶ。言い換えると split deployment の本体は「サービスを分ける」ことではなく、分けたリソース間の contract を維持することにある。
