Azure Container Apps Topology

最終更新: 2026-03-24 18:08 JST
Azure Container Apps Topology は、Project Diana の各サービスを Azure 上にどう並べるかを決めた設計図です。infra/main.bicep というファイルひとつで、API サーバー、データ取り込みジョブ、Web 画面、データ保存用テーブルをまとめて作れます。API には「取り込みジョブを動かす」ために必要な最低限の権限だけを渡します。

Azure Container Apps Topology は、Split Deployment RAG（サービスを役割ごとに分けてデプロイするやり方）を Azure の実際のサービスで実現したものです。Project Diana Agentic RAG の infra/main.bicep を見ると、API は Container App、データ取り込みは Container Apps Job、Web 画面は Static Web App という形で、それぞれ別のサービスとして動かしています。

リソース配置

Bicep テンプレートには、まず「どの環境か」「どのコンテナイメージを使うか」「検索サービスや OpenAI の接続先」「シークレット（パスワードなどの秘密情報）」を渡します。すると、diana-api-<env>、diana-ingest-<env>、diana-web-<env> という3つのサービスが作られます。さらに、データを保存するための Storage Account も作られて、ingeststatus と chathistory という2つのテーブルができます。つまり、この1つのテンプレートで、サービスもデータの保存先もまとめて管理できます。

API、データ取り込みジョブ、Web 画面、データ保存テーブルの4つを、1つのテンプレートでまとめて作ります。ジョブを動かせるのは API だけです。

API Container App

API アプリはインターネットからアクセスできるようになっていて、ポート番号は targetPort: 8000 です。使えるメモリは 1Gi、CPU は 0.5 で、常に最低1台、最大3台まで自動で増減します（minReplicas: 1、maxReplicas: 3）。パスワードや API キーなどの秘密情報として、Search のキー、OpenAI のキー、JWT の秘密鍵などが設定されています。環境変数には AZURE_CONTAINERAPPS_JOB_NAME やテーブル名も含まれます。つまりこの API アプリは、Policy-Enforced Retrieval（ルールに基づいて検索する機能）と、データ取り込みジョブを動かす機能の両方を持っています。

Manual ingest job

データ取り込み（ingest）は Microsoft.App/jobs として作られていて、必要なときに手動で動かします。最大実行時間は replicaTimeout: 900（15分）で、失敗したら1回だけやり直します（replicaRetryLimit: 1）。同時に動くのは1つだけです（parallelism: 1）。ジョブは 2Gi / 1 CPU で、API よりもパワーがあります。これは ドキュメント Ingest Pipeline（文書を読み込んで検索できるようにする処理）が重い処理だからです。Search や OpenAI のキーは API と同じものを使いますが、JWT の秘密鍵など API だけに必要なものは持っていません。

権限の絞り込み

Bicep では、API アプリに「マネージド ID」（Azure が自動で管理してくれるログイン情報）を与えます。そのとき、Azure 全体ではなく、ingest ジョブだけを操作できる権限に絞っています。権限の種類は Contributor ですが、範囲をジョブだけに限定しているので、API ができることは「取り込みジョブを動かす」ことだけです。もしもっと詳しく知りたい場合は、API-Job Trigger Boundary や Job Scope Role Assignment を見てください。

ローカルとの対応

docker-compose.yml を使うと、Azure 上の構成をローカル PC でも再現できます。chat と web は常に動くサービスとして起動します。web は chat が動いていないと使えません。ジョブは profiles: [\"job\"] というグループに入っていて、使いたいときだけ手動で起動します。この仕組みは Local Docker Compose Profile で詳しく説明しています。

ローカルとクラウドでは環境変数の値が違うので、その差を正しく設定する必要があります。環境変数ファイルの作り方は Env Profile Rendering、設定に間違いがないかの確認は Env Consistency Checks で行います。

運用上の含意

この構成では、API とジョブは同じ Storage Table を使いますが、Web 画面は Static Web App として別にデプロイされます。そのため、デプロイのときはコードだけでなく、テーブルの形、秘密情報の設定、ビルドの出力先なども合わせる必要があります。サービスを分けること自体は難しくありませんが、分けたサービス同士の「約束ごと」をきちんと守ることが大切です。

関連する話題として、Storage-Backed Status Tables と Static Web App Build Contract があります。
