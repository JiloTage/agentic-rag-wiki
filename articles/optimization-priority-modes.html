Optimization Priority Modes

最終更新: 2026-03-24 18:30 JST
Optimization Priority Modes は、Project Diana の検索処理において「品質・速度・コスト」のどれを優先するかを切り替える仕組みである。normalize_optimize_priority() が環境変数から方針を読み取り、各ラウンドの検索予算（候補数や並べ替え回数）を調整する。Retrieval Round Policy と Retrieval Budgeting の間に位置し、検索全体のリソース配分を方向づける。

Optimization Priority Modes は、Project Diana Agentic RAG のチャット検索経路において、検索の「何を優先するか」を決定する設計パターンである。Retrieval Round Policy がラウンドの進行順序を決め、Retrieval Budgeting が全体の予算枠を管理するのに対し、Optimization Priority Modes はその予算をどの方向に傾けるかを制御する。具体的には quality（品質重視）、latency（速度重視）、cost（コスト重視）の 3 つのモードがあり、Policy-Enforced Retrieval の実行時にラウンドごとの候補数や再ランキング（並べ替え）の深さを変える。

3 つのモード

normalize_optimize_priority() 関数は、環境変数 OPTIMIZE_PRIORITY の値を読み取り、以下の 3 つのモードのいずれかに正規化する。

  • quality — 品質重視。top_k_vector（ベクトル検索の候補数）と top_k_final（最終出力数）をフルに使い、再ランキングも全候補に対して行う。応答時間やコストよりも、回答の正確さを最優先する。
  • latency — 速度重視。候補数を標準の約 50% に削減し、再ランキングの深さも浅くする。ユーザーの待ち時間を短くしたい対話的なシナリオに向いている。
  • cost — コスト重視。候補数を標準の約 33% に削減し、再ランキングをほぼスキップする。大量のバッチ処理や、検索精度よりも運用コストの最小化が求められる場面で使う。

モード名が認識できない値だった場合、デフォルトで quality にフォールバックする。この挙動は Env Consistency Checks で事前に検出できるが、ランタイムでも安全側に倒すように設計されている。

予算との連携

Optimization Priority Modes は単独で動くのではなく、Retrieval Round Policy の round_budget_snapshot() の中で呼び出される。具体的な流れは次のとおりである。

  1. round_budget_snapshot() が全体予算から使用済み分を差し引き、1 ラウンドあたりの基本予算を算出する。
  2. normalize_optimize_priority() が現在のモードを返す。
  3. モードに応じた乗数（quality: 1.0、latency: 0.5、cost: 0.33）を基本予算に掛ける。
  4. 結果が top_k_vector と top_k_final として各検索呼び出しに渡される。

この仕組みにより、Retrieval Budgeting が定める全体枠の中で、モードが候補数の「密度」を変える役割を果たす。複数サブクエスチョンがある場合は Multi-Query Budget Splitting がさらに分配を調整する。

環境変数 OPTIMIZE_PRIORITY から正規化関数を経て、各モードの乗数がラウンド予算に反映されるまでの流れ。

環境変数と設定契約

OPTIMIZE_PRIORITY は 環境変数コントラクト管理 の env/contract.json に登録されるキーの一つである。owner は chat、domain は retrieval に分類される。許容値は quality | latency | cost の 3 択で、それ以外の値は Env Consistency Checks の CI ステップで検出される。

デプロイ先のプロファイルごとに異なる値を設定するには、Env Profile Rendering の profile override を使う。たとえば本番環境では quality、ステージング環境では latency、負荷テスト環境では cost といった使い分けが一般的である。

バリデーションの流れ

設定値のバリデーション（検証）は 2 段階で行われる。第 1 段階は CI 時の静的チェックで、check_env_consistency.py が contract.json と profile override の整合性を確認する。第 2 段階はランタイムの normalize_optimize_priority() で、未知の値が渡された場合に quality へフォールバックしつつ、validation_log に警告を記録する。

再ランキング深度の制御

モードが最も大きく影響するのは再ランキング（reranking）のステップである。再ランキングとは、ベクトル検索で取得した候補を、より精度の高いモデルで並べ替える処理のことである。

  • quality モード — ベクトル検索の全候補（top_k_vector 件）を再ランキング対象にする。計算コストは高いが、最も正確な順位が得られる。
  • latency モード — 上位 50% のみを再ランキング対象にする。Azure AI Search のスコアが高い候補だけに絞ることで、再ランキングの呼び出し時間を短縮する。
  • cost モード — 再ランキングをスキップし、ベクトル検索のスコア（azure_score）をそのまま最終スコアとして使う。外部 API の呼び出し回数が減るため、コストを大幅に抑えられる。

再ランキングの結果は Workspace Trace UI でラウンドごとに確認できる。Live Retrieval Lanes のリアルタイム表示でも、どのモードで動作しているかがレーンのラベルに表示される。

スコア計算への影響

最終的なドキュメントの順位は azure_score * (0.6 + 0.4 * subquestion_weight) で計算される。この計算式自体はモードによって変わらないが、入力となる azure_score の精度がモードによって異なる。quality モードでは再ランキング後の精緻なスコアが使われるのに対し、cost モードではベクトル検索の粗いスコアがそのまま使われる。

そのため、cost モードでは同じクエリでも異なるドキュメントが上位に来る可能性がある。この差異を監視するには、Priority Mode Drift Analysis（モード間のスコア差異分析）を行い、品質劣化が許容範囲内かを定期的に確認する必要がある。

システム全体での位置づけ

Optimization Priority Modes は、Policy-Enforced Retrieval の「何を検索するか」と Retrieval Round Policy の「どの順番で検索するか」の間に位置し、「どの程度のリソースを使って検索するか」を決定する。この 3 つの仕組みが連携することで、Split Deployment RAG の検索経路は柔軟かつ制御可能になる。

運用面では、モードの切り替えはデプロイ単位で行われる。リクエストごとの動的な切り替えは現在サポートされていないが、将来的には Per-Request Priority Override（リクエスト単位の優先度上書き）や Adaptive Mode Switching（負荷に応じた自動切り替え）が検討されている。また、モードの効果を定量的に評価するために Mode A/B Testing Framework（モード比較テスト基盤）の構築も議論されている。
