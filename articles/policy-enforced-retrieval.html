Policy-Enforced Retrieval

最終更新: 2026-03-24 18:08 JST
Policy-Enforced Retrieval は、「検索できる範囲」をルールで制御する仕組みです。必ず適用されるフィルターと、AIモデルが提案するヒントを組み合わせて使います。モデルは検索範囲を狭めることはできますが、広げることはできません。検索の回数や使えるリソースの量もルールで管理されます。

Policy-Enforced Retrieval は、Project Diana Agentic RAG のチャット機能で使われている検索制御の仕組みです。ResponsesOrchestrator、PolicyEngine、ToolRoundExecutor、round_policy といった部品が協力して動きます。Split Deployment RAG の API 側で、検索の流れをコントロールしています。

中心不変条件

policy.py には、この仕組みの一番大事なルールが書かれています。それは effective_filter = mandatory_filter AND validated_optional_filter というものです。つまり、「必ずかかるフィルター」と「チェック済みの追加フィルター」を合わせたものが、実際に使われるフィルターになります。AIモデルが提案するフィルターは、検索範囲を狭めることしかできず、広げることはできません。

これは、検索をモデルに丸投げしないための安全策です。たとえば user_id や category_id のような必須の制約を守らない検索は、品質以前に権限や契約の問題につながります。そのため、環境変数コントラクト管理 の設定と policy engine の実装がセットで動くようになっています。

AIモデルの提案は optional filter として検索範囲を狭めることだけができます。必須の制約（mandatory constraint）を広げたり外したりすることはできません。

リクエスト境界

/chat のルート（エンドポイント）は、まずユーザーの認証、メッセージの長さ、履歴の長さ、リクエストの信頼情報、スレッドの所有者などを確認します。その確認が終わってから _chat_policy_retrieval を呼び出します。つまり、検索のルール適用は回答を作った後ではなく、リクエストを受け取ったすぐ後から始まっています。

このとき、アプリの実行環境には OpenAI クライアント、検索アダプター、PolicyEngine、FilterCompiler、ObservabilityStore などがまとめて入っています。ルールの適用はひとつのルート（エンドポイント）だけの話ではなく、アプリ全体の実行環境の一部です。Workspace Trace UI で見えるトレース（処理の記録）も、この実行環境の情報を反映しています。

Round policy と narrowing

round_policy.py は、検索を何回行うか、その順番をどうするかを決めています。1回目は mandatory_broad（必須フィルターだけで広く検索）、2回目は optional_narrowing（追加フィルターで絞り込み）、3回目以降は optional_relaxed（条件をゆるめて検索）です。順番を飛ばすことはできないようになっています。

この順番にしている理由は、最初からAIモデルのヒントに頼りすぎないためです。まず必須フィルターだけで広く検索し、次にラベルやカテゴリを使って絞り込み、それでも足りなければ後のラウンドで条件をゆるめます。詳しくは Retrieval Round Policy をご覧ください。

Budget と ranking

ToolRoundExecutor は、検索の各ラウンドで「何件まで取得するか」という予算を計算します。検索を何回呼ぶかに応じて、候補の数やリランク（並べ替え）の数を配分します。このとき optimization_priority の設定が quality（質を重視）、latency（速さを重視）、cost（コストを重視）のどれかによって、件数が変わります。つまり、検索のたびに使えるリソースが決まっていて、予算のように管理されています。

また、質問を分割して検索した場合は、Azure のスコアと重みを掛け合わせて証拠を選び直します。検索エンジンが返した順番をそのまま使うわけではありません。この仕組みについては Retrieval Budgeting も参考になります。

Diagnostics とリリース運用

Project Diana では、ルールの設定をドキュメントにも記録しています。docs/retrieval-release-verification.md には、リリース前に確認すべきことが書かれています。具体的には、環境変数の整合性チェック、コードの静的解析、回帰テスト、証拠が少ないときの動作確認などです。特に POLICY_MANDATORY_FIELDS と REQUIRED_FILTERABLE_FIELDS は検索のルールに直接関わるので、設定を変えるときはリリース前の確認対象になります。

さらに、検索中のイベント記録、ランキングの診断、ラウンドの診断が整っています。そのため、「なぜ検索が絞り込まれたのか」「なぜフォールバック（代替手段）が使われたのか」をトレース上で追うことができます。関連する話題として Live Retrieval Lanes や Taxonomy-Driven Metadata Validation があります。

Ingest 側とのつながり

ルールの適用はチャット側の機能ですが、検索で使うヒントや証拠は ドキュメント Ingest Pipeline（文書の取り込み処理）が作ったものです。doc_categories、task_intent、chunk_keywords などは取り込み時に生成されます。そのため、検索を絞り込めるかどうかは、取り込み側で作られたメタデータの質に左右されます。つまり、検索のルールはチャット側だけの話ではなく、取り込み処理と表裏一体です。
