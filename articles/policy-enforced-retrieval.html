Policy-Enforced Retrieval

最終更新: 2026-03-24 17:53 JST
Policy-Enforced Retrieval は、Project Diana の chat 経路で mandatory filter と validated optional hint を組み合わせて retrieval を制御する設計である。モデル提案は検索範囲を狭められても広げられず、round policy と budget がその上で動く。

Policy-Enforced Retrieval は、Project Diana Agentic RAG の chat サービスが採る retrieval 制御モデルである。ResponsesOrchestrator、PolicyEngine、ToolRoundExecutor、round_policy が連携し、Split Deployment RAG の API 側で検索経路を統制する。

中心不変条件

policy.py の docstring は、この設計の核心を明示している。すなわち effective_filter = mandatory_filter AND validated_optional_filter であり、モデル提案は scope を広げず、せいぜい narrow するだけである。trusted request context から導かれる mandatory constraint が土台にあり、model-suggested metadata hint は taxonomy や allowlist を通過したものだけが追加される。

これは、検索をモデル任せにしないための境界である。たとえば user_id や category_id のような hard constraint を守らない retrieval は、品質以前に権限や契約の問題を起こす。そのため 環境変数コントラクト管理 による設定と、policy engine の実装が一体で機能している。

モデル提案は optional filter として narrow できるだけで、mandatory constraint を広げたり外したりはできない。

リクエスト境界

/chat route は、認証済みユーザー、message length、history length、trusted context、thread ownership などを先に検証したうえで _chat_policy_retrieval を呼ぶ。つまり retrieval policy は回答生成の後付けではなく、HTTP 境界のすぐ内側で始まっている。

このとき runtime には OpenAI client、Search adapter、PolicyEngine、FilterCompiler、ObservabilityStore などが束ねられており、policy enforcement は route 単体ではなくアプリケーション runtime の一部である。したがって Workspace Trace UI で見える trace も、この runtime 層の診断情報を反映している。

Round policy と narrowing

round_policy.py は retrieval round の遷移を固定している。1 回目は mandatory_broad、2 回目は optional_narrowing、それ以降は optional_relaxed である。しかも out-of-order を禁じる guard があり、いきなり緩和 round へ飛ぶことは許されない。

この round 制御の意味は、最初から metadata hint に強く依存しないことにある。初回は mandatory だけで広く取り、2 回目で accepted label / category / intent を使って narrow し、それでも不足するなら later round で緩める。これは Retrieval Round Policy として独立に読む価値がある。

Budget と ranking

ToolRoundExecutor は round ごとの top-k budget を計算し、planned retrieval call 数に応じて local candidate budget と rerank budget を配分する。ここで optimization_priority が quality、latency、cost のいずれかになり、候補件数と rerank 件数が変わる。つまり retrieval は 1 回ごとに固定コストではなく、明示的な予算消費モデルで動く。

また subquestion ごとに Azure score と weight を掛け合わせて ranked evidence を選び直しているため、検索エンジンの返却順をそのまま使っているわけでもない。ここから Retrieval Budgeting や multi-subquestion ranking を派生テーマとして切り出せる。

Diagnostics とリリース運用

Project Diana は policy 設定を docs にも落としており、docs/retrieval-release-verification.md では必須チェックとして env consistency、static analysis、regression test、low-evidence fallback の観測を要求している。特に POLICY_MANDATORY_FIELDS と REQUIRED_FILTERABLE_FIELDS は retrieval contract に直接影響するため、設定変更は release gate の対象になる。

さらに live lane event、ranking diagnostics、round diagnostics が整備されているため、retrieval がなぜ narrow されたか、なぜ fallback したかを trace 上で追える。これは Live Retrieval Lanes や Taxonomy-Driven Metadata Validation といった話題に接続する。

Ingest 側とのつながり

policy enforcement は chat 側の機能だが、受け取る hint や evidence は ドキュメント Ingest Pipeline に依存している。doc_categories、task_intent、chunk_keywords は ingest 側で生成されるため、policy が narrow できるかどうかは upstream metadata quality に左右される。つまり retrieval policy は chat 単独の話ではなく、ingest と表裏一体である。
