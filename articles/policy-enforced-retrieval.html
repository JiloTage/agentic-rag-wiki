Policy-Enforced Retrieval

最終更新: 2026-03-24 18:08 JST
Policy-Enforced Retrieval は、「検索できる範囲」をルールで決める仕組みです。必ずかかるフィルターと、AIが提案するヒントを組み合わせます。AIは検索範囲を狭めることだけができ、広げることはできません。

Policy-Enforced Retrieval は、Project Diana Agentic RAG のチャットで使っている検索ルールの仕組みです。ResponsesOrchestrator、PolicyEngine、ToolRoundExecutor、round_policy という部品が一緒に動きます。Split Deployment RAG の API 側で、検索の流れを管理しています。

中心不変条件

policy.py に一番大事なルールが書いてあります。それは effective_filter = mandatory_filter AND validated_optional_filter です。「必ずかかるフィルター」と「チェック済みの追加フィルター」を合わせたものが、実際のフィルターになります。AIが提案するフィルターは、検索を狭くすることだけできます。広くすることはできません。

これは、検索をAIに任せきりにしないための安全策です。たとえば user_id や category_id のような必須の条件を無視した検索は、権限やセキュリティの問題になります。そのため、環境変数コントラクト管理 の設定と policy engine がセットで動くようになっています。

AIの提案は optional filter として検索を狭めることだけができます。必須のフィルターを広げたり外したりすることはできません。

リクエスト境界

/chat のエンドポイント（リクエストの受け口）は、まずログインの確認、メッセージの長さ、履歴の長さ、スレッドの持ち主などを調べます。それが終わってから _chat_policy_retrieval を呼びます。検索ルールの適用は、回答を作る前の段階から始まっています。

アプリの中には、OpenAI クライアント、検索アダプター、PolicyEngine、FilterCompiler、ObservabilityStore などがまとめて入っています。ルールの適用は1つの機能だけでなく、アプリ全体にまたがっています。Workspace Trace UI で見える処理の記録も、この仕組みの情報を使っています。

Round policy と narrowing

round_policy.py は、検索を何回やるか、どんな順番でやるかを決めます。1回目は mandatory_broad（必須フィルターだけで広く検索）です。2回目は optional_narrowing（追加の条件で絞り込み）です。3回目以降は optional_relaxed（条件をゆるめて検索）です。この順番は飛ばせません。

こうする理由は、最初からAIのヒントに頼りすぎないためです。まず必須の条件だけで広く探します。次にラベルやカテゴリで絞ります。それでも足りなければ条件をゆるめます。くわしくは Retrieval Round Policy をご覧ください。

Budget と ranking

ToolRoundExecutor は、検索のたびに「何件まで取るか」を計算します。検索の回数に合わせて、取得する件数やリランク（並べ替え）の件数を割り当てます。optimization_priority が quality（質を優先）、latency（速さを優先）、cost（コストを優先）のどれかで、件数が変わります。検索に使えるリソースを予算のように管理しています。

質問を分けて検索した場合は、Azure のスコアと重みを使って、結果を並べ替えます。検索エンジンが返した順番をそのまま使うわけではありません。くわしくは Retrieval Budgeting をご覧ください。

Diagnostics とリリース運用

docs/retrieval-release-verification.md には、リリース前に確認すべきことが書いてあります。環境変数のチェック、コードの分析、テスト、結果が少ないときの動作確認などです。特に POLICY_MANDATORY_FIELDS と REQUIRED_FILTERABLE_FIELDS は検索ルールに直接関わるので、変えるときはリリース前に必ず確認します。

検索中のイベント記録、ランキングの確認、ラウンドの確認も用意されています。「なぜ検索が絞られたのか」「なぜ別の方法が使われたのか」を、あとから調べられます。関連する記事として Live Retrieval Lanes や Taxonomy-Driven Metadata Validation があります。

Ingest 側とのつながり

検索ルールはチャット側の機能ですが、検索に使うヒントやデータは ドキュメント Ingest Pipeline（文書の取り込み処理）が作ったものです。doc_categories、task_intent、chunk_keywords などは取り込みのときに作られます。検索をうまく絞れるかどうかは、取り込み側が作ったデータの質で決まります。検索ルールとデータの取り込みは、セットで考える必要があります。
