Picture OCR Stage

最終更新: 2026-03-24 18:30 JST
Picture OCR Stage は、ドキュメント内の画像から文字を読み取る処理ステージです。parse ステージで抽出された画像に対して OCR（光学文字認識）を実行し、読み取ったテキストを後続の chunk・metadata ステージへ渡します。画像に埋め込まれた表やキャプションも検索対象にできるため、retrieval の精度向上に貢献します。

Picture OCR Stage は、ドキュメント Ingest Pipeline の中で、画像に含まれる文字を読み取るステージです。PDF やスライドには、グラフ・表・スクリーンショットなど、本文とは別に画像として埋め込まれた情報が多くあります。これらの画像内テキストを見逃すと、検索時にヒットしない「見えないデータ」になってしまいます。Picture OCR Stage はこの問題を解決するために、parse ステージの直後に実行されます。Project Diana Agentic RAG では job_src.picture_ocr モジュールがこの処理を担当します。

パイプライン内の位置づけ

Picture OCR Stage は、ドキュメント Ingest Pipeline の parse ステージと chunk ステージの間に位置します。parse ステージが文書構造を解析し、本文テキストと画像を分離したあと、画像だけを対象に OCR を実行します。OCR の結果は、画像のメタデータに ocr_text フィールドとして付与されます。その後、Docling Hybrid Chunking がチャンクを作る際に、この ocr_text を search_text や context_full に組み込みます。

parse が出力した画像群だけを Picture OCR Stage が処理し、読み取ったテキストを本文と合流させて chunk へ渡します。

OCR エンジンと処理方式

Picture OCR Stage では、Azure Document Intelligence（旧 Azure Form Recognizer）の OCR API を使って画像からテキストを読み取ります。この API は、印刷文字だけでなく手書き文字や表構造も認識できます。処理の流れは次のとおりです。

  1. parse ステージが出力した画像一覧を受け取る
  2. 各画像を Azure Document Intelligence へ送信する
  3. 返ってきたテキストを、画像ごとに ocr_text フィールドへ格納する
  4. 表構造が検出された場合は、Markdown テーブル形式に変換して保持する

OCR の実行には API 呼び出しのコストと時間がかかるため、Parse Mode Fallback と連携して、必要な画像だけに OCR を適用する仕組みになっています。parse_mode=auto のとき、本文テキストが十分に抽出できたページでも、画像が含まれていれば Picture OCR Stage は実行されます。

選択的 OCR の仕組み

すべての画像に OCR をかけると、コストと時間が大きくなります。そこで Picture OCR Stage は、次の基準で OCR の要否を判定します。

  • サイズフィルタ: 極端に小さい画像（アイコン、装飾画像など）は OCR をスキップする
  • 形式フィルタ: ベクター画像や透過 PNG のうち、テキストを含まない可能性が高いものを除外する
  • 重複検出: 同一ページ内に類似の画像が複数あるとき、1つだけ OCR を実行して結果を共有する

この選択的 OCR により、全画像に OCR をかける場合に比べて処理時間とコストを大幅に削減できます。フィルタの閾値は 環境変数コントラクト管理 の設定で調整可能です。

OCR テキストの正規化

OCR が返すテキストはそのままでは検索に適さない場合があります。Picture OCR Stage は、読み取ったテキストに次の正規化処理を施します。

  • 不要な改行・空白の除去
  • 表構造の Markdown テーブルへの変換
  • 文字化けが疑われる文字列の除去（confidence スコアが閾値未満のセグメント）
  • 言語検出に基づく適切なトークン分割

正規化されたテキストは ocr_text としてチャンクに組み込まれ、Docling Hybrid Chunking で search_text の一部になります。

エラー処理と耐障害性

OCR API の呼び出しは外部サービスへの依存であるため、タイムアウトやレート制限が発生する可能性があります。Picture OCR Stage では次のエラー処理を実装しています。

  • リトライ: 一時的なエラー（429 Too Many Requests、503 など）に対して指数バックオフでリトライする
  • 部分成功: 一部の画像で OCR が失敗しても、成功した画像の結果は保持して処理を続行する
  • エラーコード: 全画像の OCR が失敗した場合は picture_ocr_total_failure を記録するが、パイプライン全体は停止しない

この「部分成功を許容する」設計は、ドキュメント Ingest Pipeline の metadata ステージと同じ思想です。OCR テキストは「あると検索精度が上がるが、なくても致命的ではない」データなので、失敗時にパイプライン全体を止めるのは過剰です。エラー情報は callback（結果通知）に含まれるため、Workspace Trace UI で事後確認できます。

チャンクへの統合

Picture OCR Stage が出力した ocr_text は、Docling Hybrid Chunking のチャンク生成プロセスで活用されます。具体的には次のように統合されます。

  • search_text: チャンクの検索用テキストに ocr_text を連結する。これにより、画像内のテキストでも全文検索がヒットする
  • context_full: チャンクの文脈情報に画像の説明として ocr_text を含める
  • picture_texts: OCR 結果の原文を保持する配列フィールド。デバッグや再処理に使う

この統合により、Policy-Enforced Retrieval でユーザーが画像内の用語を検索したとき、該当するチャンクが正しくヒットするようになります。たとえば、スライド内の図表に書かれた数値やラベルを検索キーワードとして使えます。

設定と環境変数

Picture OCR Stage の動作は、環境変数コントラクト管理 で定義された環境変数で制御します。主な設定項目は以下のとおりです。

  • AZURE_DOC_INTELLIGENCE_ENDPOINT: Azure Document Intelligence のエンドポイント URL
  • AZURE_DOC_INTELLIGENCE_KEY: 認証キー
  • PICTURE_OCR_MIN_SIZE: OCR を実行する画像の最小サイズ（ピクセル）
  • PICTURE_OCR_CONFIDENCE_THRESHOLD: OCR 結果を採用する最低信頼度スコア
  • PICTURE_OCR_TIMEOUT_SEC: 1 画像あたりの OCR タイムアウト（秒）

これらの設定は Split Deployment RAG の job プロファイルにのみ含まれ、API や Web UI のプロファイルには不要です。Env Consistency Checks で、job プロファイルに必要な設定が漏れていないかを検証できます。
