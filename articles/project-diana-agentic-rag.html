Project Diana Agentic RAG

最終更新: 2026-03-24 18:08 JST
Project Diana Agentic RAG は、policy-enforced retrieval と document ingest を分離デプロイで運用するための private repository である。FastAPI API、ingest job、Next.js ワークスペース UI、Azure Bicep 定義を単一のモノレポで束ね、実装と運用手順を同時に管理している。

Project Diana Agentic RAG は、ACES の private GitHub repository aces-inc/project-diana-agentic-rag として管理されている Agentic RAG 基盤である。README の定義では「policy-enforced KB retrieval + split deployment」が中心思想であり、Split Deployment RAG、Policy-Enforced Retrieval、環境変数コントラクト管理 を同時に扱う設計が採られている。

リポジトリの位置づけ

このリポジトリは default branch が master の monorepo であり、単純な API 実装ではなく、検索付きチャット、非同期 ingest、フロントエンド作業領域、Azure への配備定義、検証 runbook までをひとまとめにしている。トップレベル README から見える責務は apps/chat、apps/job、apps/web、infra の 4 つに整理されており、単一プロセスではなく役割分離を前提にした運用を意識している。

特徴的なのは、RAG の実装だけでなく運用規約も同じ repo に置かれている点である。たとえば docs/split-local-dev.md は local runbook を、docs/env-unified-management.md は設定ガバナンスを、env/contract.json は各モジュールの環境変数契約を担う。したがってこの repo は「アプリコードの置き場」だけではなく、開発・検証・ロールアウトの共通仕様書でもある。

全体アーキテクチャ

構成は split deployment 型で、ユーザー体験を持つ apps/web、API 制御層の apps/chat、非同期 ingest を担当する apps/job が明確に分かれている。さらに Azure Search、Azure OpenAI、Document Intelligence、Storage などの周辺サービスが Bicep 定義と env contract を通じて結び付けられている。これは、検索応答と文書取り込みを同じサーバーに押し込まず、責務ごとにスケールと失敗境界を分けるための構成と読める。

Project Diana Agentic RAG の split deployment 構成。env contract が web/chat/job を横断し、chat と job が Azure サービス群を分担して利用する。

主要モジュール

Chat API

apps/chat は FastAPI ベースの API サービスであり、chat_src.api.app から auth、files、threads、chat、chat_stream、ingest、observability の各 router が組み込まれている。CORS、correlation and observability middleware、統一エラーペイロード、retryable 判定が入口で揃えられているため、RAG 本体よりも前段の API 契約と運用観測性に強く寄っていることが分かる。

依存関係には fastapi、openai、azure-search-documents、azure-ai-agents、Table/Blob Storage 系ライブラリが含まれており、検索結果の取得だけでなく認証、ファイル状態、履歴保存、ingest 起動まで同じサービス境界で扱う構成である。strict 対象に orchestrator や runtime support 群が列挙されている点からも、Policy-Enforced Retrieval をアプリケーションの中核に置いていることが読み取れる。

Ingest Job

apps/job は文書投入を非同期実行する worker であり、job_src.entrypoint では validate、prepare、parse、picture OCR、chunk、metadata、index、callback という段階的な pipeline が明示されている。特に parse と OCR を分け、chunking と metadata 生成を並列に実行し、最後に index 書き込みと callback 配信へ流す設計は、ドキュメント Ingest Pipeline を独立サービスとして切り出した理由をよく示している。

依存には docling、pypdf、tiktoken、torch、azure-search-documents、openai が含まれる。これは単なる PDF 分割ではなく、OCR、分類、埋め込み、検索用 enriched chunk の生成まで job 側で完結させる意図を表す。エラー分類コードや degrade 可能な metadata 失敗の扱いも実装されており、運用上の失敗モードを最初から想定している。

Web Workspace

apps/web は Next.js 16 と React 18.3 を使った workspace UI であり、app/page.tsx は WorkspaceShell を中心に構成される。Shell には AuthCard、ChatPanel、FilesPanel、IngestPanel、ThreadPanel、TraceModal がぶら下がっており、単一のチャット画面ではなく、認証済みユーザーが知識ベース管理と会話を横断する作業領域として設計されている。

README と local runbook が示す「streaming trace panels」という表現どおり、この UI は回答本文だけでなく retrieval、reasoning、citation を追える観測面を持つ。したがって Workspace Trace UI は単なる見た目の話ではなく、agentic retrieval の説明責任とデバッグ効率を担う要素である。

Infra

infra/main.bicep は Azure を前提に、Container Apps の API アプリ、manual trigger の ingest job、Static Web App、Storage Account、Table Storage、job 起動用 role assignment を定義している。API と job が別の実行単位として構成され、API 側には job 起動権限だけをスコープ限定で与えているため、アーキテクチャ図だけでなく権限境界も split deployment の考え方に沿っている。

この Bicep は Azure Search と Azure OpenAI の接続情報、knowledge base 名、JWT と webhook secret などを secure parameter として受け取り、実行環境へ注入する。つまりリポジトリの infra 層は、RAG 推論基盤と業務 API をつなぐ配線図であると同時に、デプロイ時の責務分担をコード化した部分でもある。

設定と運用モデル

Env Contract

この repo を特徴づける最大の仕組みの一つが env/contract.json を source of truth に据えた env governance である。ここには key の owner、domain、説明、module 別の required/non-local-required/optional ルール、さらに local、dev、stg、prod の profile override が整理されている。ドキュメントは、module catalog、.env.example、contract の 3 者を一致させることを明示しており、環境変数コントラクト管理 そのものが実装の一部になっている。

重要なのは、Python サービスが runtime で load_dotenv() を使わず、process environment のみを読むと明記している点である。これは import 時の副作用を避けつつ、CI 検証と local override を両立するための方針であり、設定読み出しの一貫性を高めている。

Local Validation とロールアウト

起動方法は Docker あり・なしの両方が README に整理されているが、推奨は docker compose up --build を使う local-first 運用である。API は /health と /ready を持ち、web は port 3000、chat は port 8000 を基本にする。テスト面では chat/job の pytest、web の vitest と Playwright、env convention/consistency check、さらに OpenSpec validation が用意されているため、実装・契約・UI の三層で検証する体制になっている。

docs/split-local-dev.md には ROLLOUT_PHASE を用いた段階的な release gate も記されており、Retrieval dependency health や phase 4 migration rollback まで手順化されている。つまりこの repo は PoC 的な RAG 実験場ではなく、段階的デプロイと失敗時の後退戦略を持つ運用対象として設計されている。

読み解くポイント

この repo を理解するときは、単に「どのモデルを呼んでいるか」を追うだけでは不十分である。むしろ次の 4 観点で読むと構造が見えやすい。

  • chat と job を分けることで、対話応答と文書処理の失敗境界をどのように切っているか。
  • env contract によって、モジュールごとの設定責務をどこまで機械的に検証しているか。
  • trace UI と observability route により、検索根拠と推論経路をどこまで可視化しているか。
  • runbook、phase rollout、infra 定義まで含めて、RAG システムを「継続運用する製品」として扱っているか。

次に掘るべき記事候補としては、Split Deployment RAG、ドキュメント Ingest Pipeline、Workspace Trace UI、Policy-Enforced Retrieval、環境変数コントラクト管理 が自然である。これらを分解すると、Project Diana Agentic RAG の設計思想をサービス境界、設定、UI、運用の各面から再構成できる。
