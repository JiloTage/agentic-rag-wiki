Project Diana Agentic RAG

最終更新: 2026-03-24 18:08 JST
Project Diana Agentic RAG は、ACES が作っている RAG（質問に答えるとき、関連する文書を自動で探してきて使う仕組み）のプロジェクトです。API サーバー、文書の取り込み、Web 画面、Azure へのデプロイ設定が、すべて 1 つのリポジトリにまとまっています。決められたルールに従って情報を検索する仕組みになっていて、それぞれのサービスを別々に動かせるように設計されています。

Project Diana Agentic RAG は、ACES の GitHub リポジトリ aces-inc/project-diana-agentic-rag で開発されています。このシステムには 2 つの大きな特徴があります。1 つ目は「ルールに基づいて検索する」こと、2 つ目は「サービスを分けてデプロイする」ことです。Split Deployment RAG、Policy-Enforced Retrieval、環境変数コントラクト管理 という 3 つの考え方をベースにしています。

リポジトリの位置づけ

このリポジトリは、メインブランチが master のモノレポ（複数のプロジェクトを 1 つのリポジトリにまとめたもの）です。チャット API、文書の取り込み、Web 画面、Azure のデプロイ設定、テスト手順書まで、全部ここに入っています。フォルダは大きく apps/chat、apps/job、apps/web、infra の 4 つに分かれていて、それぞれ別のサービスとして動きます。

面白いのは、コードだけでなく運用のルールもこのリポジトリに入っていることです。たとえば docs/split-local-dev.md にはローカル環境の立ち上げ方が書いてあります。docs/env-unified-management.md には設定の管理ルールがあります。env/contract.json には、どのモジュールにどの環境変数が必要かの一覧があります。つまり、このリポジトリはコードの保管場所であると同時に、開発チームのルールブックでもあるのです。

全体アーキテクチャ

システムは「分離デプロイ」という形で動いています。ユーザーが使う画面（apps/web）、API サーバー（apps/chat）、文書の取り込み処理（apps/job）が、それぞれ別のサービスとして独立しています。Azure Search や Azure OpenAI などの外部サービスとは、Bicep（Azure のインフラ設定ファイル）と環境変数でつながっています。この仕組みのおかげで、チャットと文書処理を別々に増やしたり減らしたりできます。また、片方が止まっても、もう片方への影響が少なくなります。

Project Diana Agentic RAG のサービス構成図です。環境変数の設定ファイル（env contract）が web・chat・job の 3 つで共有されています。chat と job は、それぞれ必要な Azure サービスを使います。

主要モジュール

Chat API

apps/chat は FastAPI で作られた API サーバーです。chat_src.api.app の中に、いくつかの機能がまとまっています。ログイン認証（auth）、ファイル管理（files）、会話の管理（threads）、チャット（chat、chat_stream）、文書の取り込み開始（ingest）、そして動作の監視（observability）です。リクエストを受け取るところで、エラーの形式をそろえたり、再試行の判定をしたりしていて、使いやすさと運用のしやすさを大切にしています。

主なライブラリは fastapi、openai、azure-search-documents、azure-ai-agents などです。この API サーバーは検索だけでなく、認証やファイル管理、会話の保存、文書取り込みの開始も担当します。特に大事なのは Policy-Enforced Retrieval（決められたルールに従って検索する仕組み）で、これがアプリの中心になっています。

Ingest Job

apps/job は、文書を取り込む処理を裏側で実行するプログラムです。job_src.entrypoint の中では、次の順番で処理が進みます。まずファイルの検証、次に準備、解析、画像の OCR（画像から文字を読み取ること）、文書の分割、メタデータの追加、検索インデックスへの登録、そして完了の通知です。このように多くのステップがあるため、ドキュメント Ingest Pipeline として独立したサービスに分けられています。

使っているライブラリは docling、pypdf、tiktoken、torch、azure-search-documents、openai などです。PDF を分割するだけではなく、OCR で文字を読み取ったり、内容を分類したり、検索できるようにデータを整えたりと、幅広い処理を行います。エラーが起きても全体を止めずに続ける仕組みも入っています。

Web Workspace

apps/web は Next.js 16 と React 18.3 で作られた Web 画面です。メインページ app/page.tsx には、WorkspaceShell という大枠の中にいくつかの画面パーツが入っています。ログイン画面（AuthCard）、チャット画面（ChatPanel）、ファイル一覧（FilesPanel）、取り込み状態の表示（IngestPanel）、会話一覧（ThreadPanel）、処理の追跡表示（TraceModal）です。ただのチャット画面ではなく、文書の管理と会話をまとめてできる作業スペースになっています。

この画面では、AI の回答だけでなく「どの文書を探したか」「どう考えたか」「どこを引用したか」もその場で見ることができます。Workspace Trace UI はデザインの話ではなく、AI がなぜその回答をしたのかを確認したり、問題を調べたりするための大事な機能です。

Infra

infra/main.bicep は、Azure 上のインフラ（サーバーやストレージなど）をコードで書いた設定ファイルです。API 用の Container App、文書取り込み用の job、Web 画面用の Static Web App、データ保存用の Storage Account などが書かれています。API と job は別々に動くようになっていて、API には「job を起動する権限だけ」が与えられています。権限の面でも、サービスをきちんと分ける考え方が守られています。

この Bicep ファイルには、Azure Search や Azure OpenAI への接続先、ナレッジベースの名前、認証に使う秘密鍵なども安全な形で渡されます。つまり、このファイルは「どのサービスをどうつなぐか」の設計図であり、デプロイするときの役割分担をコードとして残したものです。

設定と運用モデル

Env Contract

このリポジトリで特に大事な仕組みが、env/contract.json を使った環境変数の管理です。このファイルには、それぞれの環境変数について「誰が管理するか」「どんな種類か」「何のためか」「どのモジュールで必要か」が書かれています。また、local、dev、stg、prod といった環境ごとの値も入っています。ルールとして、モジュール一覧・.env.example・contract の 3 つは常に合わせておく必要があります。こうした 環境変数コントラクト管理 が、開発の進め方の一部になっています。

もう 1 つ注目したいのは、Python のサービスでは load_dotenv() を使わず、OS の環境変数だけを読むようにしていることです。こうすると、コードを動かしたときに予想外の動きが起きにくくなります。CI でのテストと、ローカルでの設定変更を両立しやすくなるメリットもあります。

Local Validation とロールアウト

起動方法は、Docker を使う場合と使わない場合の両方が README に書かれています。おすすめは docker compose up --build を使う方法です。API には /health と /ready という状態確認の仕組みがあります。web は port 3000、chat は port 8000 で動きます。テストは 3 種類あります。chat と job のテストには pytest、web のテストには vitest と Playwright、設定の整合性チェックには env のチェックツールを使います。コード・設定・画面の 3 つの面からテストできる体制です。

docs/split-local-dev.md には、ROLLOUT_PHASE を使って段階的にリリースする手順も書かれています。問題が起きたときに前のバージョンに戻す方法まで手順化されています。つまり、このリポジトリは実験的な RAG ではなく、少しずつリリースして、問題があれば戻せる「本番運用を前提としたシステム」として作られています。

読み解くポイント

このリポジトリを理解するには、「どの AI モデルを使っているか」を見るだけでは足りません。次の 4 つの視点で読むと、全体の仕組みがわかりやすくなります。

  • chat と job を分けることで、チャットの応答と文書処理のエラーをどう切り離しているか。
  • env contract を使って、モジュールごとの設定を自動でチェックできるようにしているか。
  • trace UI や監視の仕組みで、「なぜその検索結果を返したか」をどこまで見えるようにしているか。
  • 運用手順書、段階的リリース、インフラ定義まで含めて、RAG を「ずっと使い続ける製品」として扱っているか。
