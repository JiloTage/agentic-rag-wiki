Retrieval Round Policy

最終更新: 2026-03-24 18:08 JST
Retrieval Round Policy は、Project Diana の chat orchestrator が retrieval round を mandatory_broad → optional_narrowing → optional_relaxed の順に進める制御である。accepted metadata hint と残予算を見ながら round budget を切り、out-of-order な緩和を防ぐ。

Retrieval Round Policy は、Policy-Enforced Retrieval を実際の retrieval call 列へ落とし込む運用則である。Project Diana Agentic RAG の chat orchestrator は、round 数、accepted metadata hint、残予算をもとに毎回の検索条件を決め、Split Deployment RAG の API 経路で無秩序な filter 緩和が起きないよう固定している。

Round の並び順

resolve_round_policy() は round 数から適用名を決める。1 回目は mandatory_broad で optional hint を使わず、2 回目は optional_narrowing で accepted label / category / intent を投入し、3 回目以降は optional_relaxed へ移る。つまり最初は mandatory constraint だけで広く取り、次に狭め、最後に再び緩める。

この順序は、metadata hint を最初から強く信じすぎないためのものである。hint が欠落していても round 1 は成立し、hint が reject された場合は optional_hints_rejected_or_missing を理由として保持したまま round 2 を進める。hint の質は ドキュメント Ingest Pipeline と downstream metadata quality に依存するため、round policy は ingest 側の品質変動も吸収する役目を持つ。

round 数だけで段階を決めるのではなく、許可された遷移順そのものを固定して retrieval を制御する。

順序ガード

allowed_next は前回適用名から許される次状態を定義している。たとえば mandatory_broad の直後に進めるのは、同じ mandatory_broad か optional_narrowing だけであり、いきなり optional_relaxed へ飛ぶことは violation になる。このため「結果が少ないからすぐ緩める」という ad-hoc 分岐が入り込みにくい。

このガードは Round Order Violation Guards として独立記事化できるほど重要で、検索品質よりも先に policy stability を守る。特に mandatory constraint を伴う system では、round 制御の逸脱は ranking の揺らぎではなく契約違反へ直結する。

Round budget

round_budget_snapshot() は top_k_vector、top_k_final、planned_retrieval_calls、validation_log を使って、その round で使える candidate budget と rerank budget を切り出す。まず global budget を max_tool_rounds 分だけ確保し、すでに消費した分を引いてから、その時点の per-query budget を計算する。

さらに normalize_optimize_priority() により priority は quality / latency / cost の三択へ正規化される。latency は候補数を半分程度まで絞り、cost は三分の一程度まで圧縮する。つまり round policy は単なる順序制御ではなく、Retrieval Budgeting と Optimization Priority Modes を伴う資源配分ルールでもある。

この設定は env 経由で runtime に入るため、誤った priority や round 数を見逃さないことも重要になる。したがって round policy の安定性は 環境変数コントラクト管理 と Env Consistency Checks の整備に支えられている。

Tool round executor への落とし込み

ToolRoundExecutor は pending retrieval を集めて非同期に dispatch し、結果の消費は元の tool-call 順に戻す。各 query には validation_log、accepted / rejected metadata、effective filter fingerprint、budget snapshot が残され、後段の answer pass や diagnostics に引き渡される。このため round policy は pure function では終わらず、実行系と監査ログの両方に書き込まれる。

結果が返ると executor は evidence を dedup し、azure_score * (0.6 + 0.4 * subquestion_weight) で再ランキングして selected references を作る。ここで複数 subquestion にまたがる予算配分が効いてくるため、Multi-Query Budget Splitting の観点でも読む価値がある。

Fallback の意味

round policy には主に三種類の fallback がある。accepted hint が空のときの optional_hints_rejected_or_missing、later round への移行理由である later_round_relaxation、そして残予算が尽きたときの round_budget_exhausted である。いずれも silent fallback ではなく validation log と query result に残る。

この設計により、trace 上で「なぜ narrow されなかったか」「なぜ later round に進んだか」を説明できる。観測面では Workspace Trace UI や Live Retrieval Lanes が、そのまま round policy の可視化レイヤーになる。

システム全体での位置づけ

round policy は Policy-Enforced Retrieval の一部だが、実際には ingest で付与された metadata、runtime env の設定、observability まで含む横断的な制御層である。したがって retrieval の誤作動を調べるときは、policy article だけでなく、ドキュメント Ingest Pipeline、Env Consistency Checks、Taxonomy-Driven Metadata Validation を併せて見る必要がある。
