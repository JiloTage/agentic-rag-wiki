Retrieval Round Policy

最終更新: 2026-03-24 18:08 JST
Retrieval Round Policy は、Project Diana のチャット機能が文書を検索するときの「進め方のルール」です。まず mandatory_broad（広く探す）、次に optional_narrowing（絞り込む）、最後に optional_relaxed（条件をゆるめる）の順に進みます。ヒント情報や残りの検索回数を見ながら各ステップの上限を決め、順番を飛ばさないようにしています。

Retrieval Round Policy は、Policy-Enforced Retrieval（ポリシーに従って検索を制御する仕組み）の中で、「どの順番で検索を呼び出すか」を決めるルールです。Project Diana Agentic RAG のチャット機能は、検索の回数、メタデータヒント（文書に付いた分類タグのこと）、残りの検索回数をもとに条件を決めます。Split Deployment RAG の API では、このルールのおかげで勝手にフィルタがゆるめられるのを防いでいます。

Round の並び順

resolve_round_policy() は、検索が何回目かに応じて、どのルールを使うかを決めます。1 回目は mandatory_broad です。ヒント情報は使わず、広く検索します。2 回目は optional_narrowing です。使えると判断されたラベルやカテゴリで絞り込みます。3 回目以降は optional_relaxed になり、条件を少しゆるめます。

この順番にしている理由は、ヒント情報を最初から信じすぎないためです。ヒントがなくても 1 回目の検索は問題なく動きます。ヒントが使えないと判断された場合は、optional_hints_rejected_or_missing という理由を記録して 2 回目に進みます。ヒントの正確さは ドキュメント Ingest Pipeline（文書を取り込む処理）の出来に左右されます。そのため、ラウンドポリシーは取り込み側の品質のばらつきも吸収する役割を持っています。

検索の回数だけで段階を決めるのではなく、「次にどの段階に進めるか」の順番をあらかじめ固定して検索を制御します。

順序ガード

allowed_next は、「今の段階のあとに進んでよい次の段階」を決めています。たとえば mandatory_broad のあとに進めるのは、同じ mandatory_broad か optional_narrowing だけです。いきなり optional_relaxed に飛ぶとエラー（violation）になります。こうすることで、「結果が少ないからすぐ条件をゆるめよう」という場当たり的な動きを防いでいます。

このガード（安全装置）は Round Order Violation Guards という別の記事にするほど大切な仕組みです。検索の質よりも先に、ルールどおりに動いているかを守ります。特に必須の制約があるシステムでは、順番の逸脱は検索結果のブレではなく、ルール違反に直結します。

Round budget

round_budget_snapshot() は、top_k_vector、top_k_final、planned_retrieval_calls、validation_log といった値を使って、そのラウンドで使える検索の「予算」（候補の数や並べ替えの回数）を計算します。まず全体の予算を max_tool_rounds の回数分だけ確保し、すでに使った分を引いてから、1 回の検索あたりの予算を出します。

さらに normalize_optimize_priority() によって、優先する方針が quality（品質重視）/ latency（速度重視）/ cost（コスト重視）の 3 つに整理されます。速度重視なら候補数を半分くらいに減らし、コスト重視なら 3 分の 1 くらいに減らします。つまりラウンドポリシーは順番を決めるだけでなく、Retrieval Budgeting（検索の予算管理）や Optimization Priority Modes（最適化の方針）と連携して、リソースの配分も行います。

これらの設定は環境変数としてアプリに渡されます。そのため、間違った方針やラウンド数が設定されないようにすることが大切です。ラウンドポリシーの安定性は、環境変数コントラクト管理 と Env Consistency Checks がしっかり整備されていることで支えられています。

Tool round executor への落とし込み

ToolRoundExecutor は、まだ実行されていない検索をまとめて同時に実行し、結果を元の呼び出し順に並べ直します。それぞれの検索には、validation_log（検証の記録）、使われたヒント・使われなかったヒント、フィルタの内容、予算の記録が残ります。これらは、あとの回答生成やトラブル調査に使われます。つまりラウンドポリシーは計算して終わりではなく、実行と記録の両方に関わっています。

検索結果が返ってくると、executor は重複する結果を取り除きます。そして azure_score * (0.6 + 0.4 * subquestion_weight) という計算式で並べ替えて、参考にする文書を選びます。複数の質問にまたがる予算の配分がここで効いてくるため、Multi-Query Budget Splitting も合わせて読むと理解が深まります。

Fallback の意味

ラウンドポリシーには、主に 3 種類のフォールバック（代替の動き方）があります。ヒントが使えなかったときの optional_hints_rejected_or_missing、次のラウンドに進む理由を示す later_round_relaxation、そして検索の予算を使い切ったときの round_budget_exhausted です。どれも黙って切り替わるのではなく、検証ログと検索結果の両方に記録が残ります。

この仕組みのおかげで、あとから「なぜ絞り込みが行われなかったのか」「なぜ次のラウンドに進んだのか」を調べることができます。Workspace Trace UI や Live Retrieval Lanes を使えば、ラウンドポリシーの動きを画面上で確認できます。

システム全体での位置づけ

ラウンドポリシーは Policy-Enforced Retrieval の一部ですが、実際には文書の取り込み時に付けられたメタデータ、環境変数の設定、動作の監視まで関わる横断的な仕組みです。そのため、検索がうまく動かないときは、この記事だけでなく、ドキュメント Ingest Pipeline、Env Consistency Checks、Taxonomy-Driven Metadata Validation（分類ルールに基づくメタデータの検証）も合わせて確認してください。
