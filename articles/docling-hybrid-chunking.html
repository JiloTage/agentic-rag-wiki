Docling Hybrid Chunking

最終更新: 2026-03-24 17:53 JST
Docling Hybrid Chunking は、Project Diana の ingest job が文書を retrieval 向けの構造化チャンクへ変換する中心処理である。単純な固定長分割ではなく、見出し、context_full、検索要約、picture OCR 由来テキストを束ねた enriched chunk を作る。

Docling Hybrid Chunking は、ドキュメント Ingest Pipeline の chunk stage に相当する実装であり、job_src.chunking を中心に組み立てられている。Project Diana Agentic RAG 全体では、この stage が生文書を Policy-Enforced Retrieval で使いやすい証拠へ変換する橋渡しになる。

なぜ Hybrid Chunking なのか

この実装が目指しているのは、文書をただ短く刻むことではない。検索時に使う短い要約、回答時に使う広めの文脈、見出しチェーン、chunk keywords、picture OCR で補われた断片までまとめて 1 つの retrieval 単位にすることが目的である。したがって chunking は前処理ではあるが、実際には検索品質を左右するモデル化工程である。

固定トークンの sliding window であれば実装は簡単だが、見出し境界や table のまとまりを壊しやすい。Project Diana は Docling の HybridChunker を基盤にし、構造を優先しながら後段で oversize split を補助する方針を採る。これは Split Deployment RAG の job 側で、計算コストを払ってでも retrieval 素材を整える選択だと言える。

Docling Hybrid Chunking は、見出し補完、HybridChunker、OCR 割当、oversize split を経て enriched chunk を生成する。

構造情報をどう保持するか

実装はまず、見出しが存在しない文書に synthetic root heading を挿入する。これにより見出しベースの文脈抽出が常に成立しやすくなる。続いて HybridChunker が文書構造を使って raw chunk を作り、後段では search_headings として見出しチェーンを保つ。

この設計の狙いは、回答生成で引用を組み立てる際に section 文脈を失わないことにある。特に Context Full Construction のような派生テーマで重要になるが、chunk の短い本文だけではなく、その周囲の節構造を追跡できることが retrieval の品質を安定させる。

検索用の enrichment

この chunk stage は raw text を返して終わらない。OpenAI を用いた要約生成により search_text を作り、chunk keywords も併せて抽出する。つまり chunk は「文書の断片」ではなく「検索システムに渡すための説明付き断片」へ変換される。

また context_full は Markdown export を元に heading boundary まで前後を拡張し、必要なら sentence boundary で切り詰める。これは全文を突っ込むのでも、断片だけに頼るのでもなく、回答生成に使うちょうど中間の粒度を作る工夫である。ここから Search Summary Enrichment や Context Full Construction を掘る余地がある。

Picture OCR との結合

chunking 実装は Picture OCR Stage の結果を chunk-local に割り当てる。重要なのは、OCR テキストを文書全体へ均等にばらまくのではなく、doc item id、ref、page などを手がかりに「この画像はどの chunk に属するか」を推定している点である。これにより図表や埋め込み画像のラベルが、関係の薄い chunk へ汚染的に混ざりにくい。

また oversize split が入る場合でも、chunk-level OCR を先頭の派生 chunk にだけ残す設計になっている。これは OCR テキストの重複によって検索ランキングが歪むのを避けるための処理であり、検索素材のノイズ制御として合理的である。

Oversize split と fallback

構造優先の chunking だけでは、巨大な table や長文 section に対処しきれない。そこで実装は token 閾値を超えた chunk を検出し、table なら header を保ちながら行単位で、通常テキストなら sentence boundary で分割する。これが Chunk Oversize Splitting に対応する部分である。

さらに parse 前段では Parse Mode Fallback があり、fast parse と OCR parse の切り替えが起こる。つまり chunk stage 単体ではなく、parse mode と OCR quality の選択が前提条件になっている。

Retrieval への含意

最終的にこの stage の成果物は、検索時に候補 evidence として扱われ、回答時に引用可能な reference へ再利用される。したがって chunk quality は単に recall の問題ではなく、引用の自然さ、回答の groundedness、ランキングの安定性に直結する。ここで作られた enriched chunk が Policy-Enforced Retrieval の入力になるため、chunking と retrieval は別工程でも密接に結び付いている。
