Job Scope Role Assignment

最終更新: 2026-03-24 18:45 JST
Job Scope Role Assignment は、Azure Container Apps 環境において API のマネージド ID に対し、ingest job リソースだけを操作できる Contributor ロールを割り当てる設計パターンである。Bicep の roleAssignments リソースでスコープを job 単体に絞ることで、最小権限の原則（Principle of Least Privilege）を実装する。

Job Scope Role Assignment は、Azure Container Apps Topology で定義される API アプリのマネージド ID（Azure が自動管理する認証情報）に対し、ingest job リソースだけを操作できる権限を与える設計である。Split Deployment RAG では API・ジョブ・Web を分離してデプロイするが、分離しただけでは権限も分離されたことにはならない。API が Azure サブスクリプション全体の Contributor 権限を持ってしまうと、ジョブ以外のリソースも変更できてしまう。そこで Bicep の Microsoft.Authorization/roleAssignments リソースの scope プロパティを使い、権限の適用範囲を ingest job リソース1つに限定する。

最小権限の原則と Azure RBAC

Azure の RBAC（Role-Based Access Control、役割ベースのアクセス制御）では、「誰が」「何を」「どこに」できるかを3つの要素で定義する。「誰が」は principalId（マネージド ID の ID）、「何を」は roleDefinitionId（Contributor などのロール定義）、「どこに」は scope（適用範囲）である。

Project Diana Agentic RAG の API は、ドキュメント Ingest Pipeline を起動するためにジョブの開始権限が必要である。しかし、Storage Account の削除や他の Container App の設定変更までできてしまっては危険である。scope を job リソースだけに絞ることで、API の「できること」を「ingest ジョブの起動・停止」だけに限定できる。

Bicep による実装

Bicep テンプレートでは、まず API の Container App にシステム割り当てマネージド ID（identity: { type: 'SystemAssigned' }）を設定する。次に、Microsoft.Authorization/roleAssignments リソースを job のスコープ内で定義する。具体的には以下の要素を指定する。

scope の指定方法

roleAssignment リソースの scope プロパティに、ingest job リソースの完全な ID を渡す。Bicep では ingestJob.id のように参照する。これにより、この roleAssignment は job リソースだけに有効になる。リソースグループ全体やサブスクリプションに対する権限は一切付与されない。

ロール定義の選択

Contributor ロール（b24988ac-6180-42a0-ab88-20f7382dd24c）は、リソースの読み取り・書き込み・削除が可能なロールである。scope を job に絞っているため、実質的にはジョブの起動（start アクション）と停止だけができる。より厳密に制御したい場合は、カスタムロール定義（Custom Role Definition Patterns）を作成して Microsoft.App/jobs/start/action だけを許可する方法もある。

権限スコープの構造

API のマネージド ID に付与される Contributor ロールは、ingest job リソースだけにスコープされる。Storage Account や Static Web App は操作できない。

API-Job Trigger Boundary との関係

Job Scope Role Assignment は権限の「設定」側であり、API-Job Trigger Boundary は権限を「使う」側の設計である。API が POST /api/jobs/{jobName}/start を呼ぶとき、Azure はまず roleAssignment を確認し、API のマネージド ID が当該 job に対して Contributor（または start アクション）を持っているかを検証する。この2つの設計が組み合わさることで、「API だけが」「ingest job だけを」起動できるという制約が実現する。

環境変数コントラクト管理 で定義される AZURE_CONTAINERAPPS_JOB_NAME は、API がどのジョブを起動するかを指定する環境変数である。この値が roleAssignment の scope と一致していなければ、実行時に権限エラーになる。Env Consistency Checks でこの整合性を事前に検証することが推奨される。

セキュリティ上の考慮事項

過剰権限のリスク

scope をリソースグループレベルに設定してしまうと、API は同じリソースグループ内のすべてのリソース（Storage Account、他の Container App、ネットワーク設定など）を変更できてしまう。Azure Container Apps Topology ではすべてのリソースが同一リソースグループに配置されるため、scope の設定ミスは特に危険である。

ID のローテーション

システム割り当てマネージド ID は Container App の再作成時に自動的に新しくなる。そのため、roleAssignment も Bicep テンプレート内で宣言的に定義しておく必要がある。手動で Azure Portal から追加した roleAssignment は、再デプロイ時に孤立する（orphaned assignment）可能性がある。Bicep Idempotent Role Binding パターンを使うことで、べき等な（何度実行しても同じ結果になる）ロール割り当てを保証できる。

監査とモニタリング

Azure Activity Log には、roleAssignment の作成・変更・削除が記録される。さらに、ジョブの起動操作も記録されるため、「誰が（どの ID が）いつジョブを起動したか」を追跡できる。これは Role Assignment Audit Trail として体系化できる。

デプロイ時の適用フロー

Bicep テンプレートのデプロイ時、リソースは依存関係に従って作成される。roleAssignment は API の Container App と ingest job の両方が作成された後に適用される。Bicep の dependsOn は暗黙的に解決されるが、明示的に書くこともできる。

デプロイの全体像は Split Deployment RAG を参照。ローカル環境での検証には Local Docker Compose Profile が使える。ただし、Docker Compose にはマネージド ID の仕組みがないため、ローカルでは環境変数による認証フォールバックが必要になる。この差異は Env Profile Rendering でプロファイルごとに吸収する。
